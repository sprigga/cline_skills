# S5F4 - Enable/Disable Alarm Ack

**Direction:** Sent by Equipment Only

---

| **S5F4** | Enable/Disable Alarm Ack | Sent by Equipment Only |

## 用意與目的

S5F4 是設備對 Host S5F3R（Enable/Disable Alarm Send）的確認回覆。Host 發出警報啟用/停用指令後，設備必須在 T3 時間內以 S5F4 告知操作是否成功。

### 在 GEM 標準中的角色

```
Host                               Equipment
  │                                    │
  │  S5F3R (Enable/Disable Alarm)  ──▶ │  ← Host 主動發送
  │                                    │    設備執行 Enable/Disable 操作
  │  S5F4  (Enable/Disable Alarm Ack) ◀│  ← 設備確認（ACKC5=0）
  │                                    │
```

S5F4 是整個警報管理指令的「閉環確認」，與 S5F2（警報通知的確認）結構完全相同，都是單一 `B:1 ACKC5`，但方向相反——S5F4 由**設備發出**，Host 接收。

### 與 S5F2 的比較

| 訊息 | 方向 | 對應請求 | 用途 |
|------|------|----------|------|
| S5F2 | Host → Equipment | S5F1R | Host 確認收到警報通知 |
| S5F4 | Equipment → Host | S5F3R | 設備確認警報啟用/停用成功 |

兩者格式相同（`B:1 ACKC5`），但語意角色相反。

### 訊息結構特點

S5F4 與 S5F2 相同，是**不使用 List 包裝**的最簡格式：根 Item 直接是 `B:1`，不是 `L:1 { B:1 }`。

## Format (TSN)

```
[ACKC5](http://hume.com/secs/items.html#ACKC5)
```

### Format Explanation

- **`[ACKC5]`** — Acknowledge Code for Stream 5，`B:1` 格式（1 byte binary）。
  - `0` = OK（成功啟用/停用 alarm）
  - 其他值 = 錯誤或拒絕

這是 Equipment 回覆 Host S5F3R（Enable/Disable Alarm Send）的確認訊息，結構與 S5F2 相同，都是單一 `ACKC5` data item。

### ACKC5 數值對照表

| 值 | 意義 |
|----|------|
| `0` (`0x00`) | OK — 成功啟用/停用 alarm |
| 非零 | 拒絕或錯誤（GEM 標準未細分） |

> **實務說明：** 設備幾乎總是回覆 `ACKC5=0`。若設備不支援指定的 ALID，部分實作會回非零值；若格式錯誤，設備可能送 S9F7 而非 S5F4。

### 範例

| 傳送內容 | 意義 |
|---------|------|
| `B:1 0` | 成功啟用/停用 alarm（OK） |
| `B:1 1` | 拒絕或錯誤 |

## C# 解析程式碼（secs4net v2）

本專案為 **Host 端**，S5F4 是發出 S5F3R 後**直接從 `SendAsync` 回傳值取得**的回覆，不需要額外的 `IMessageHandler`。這是 secs4net v2 的 request/reply 自動配對機制。

### 解析流程

1. **`SendAsync` 回傳 reply** — `secsGem.SendAsync(s5f3)` 內部等待對應 Transaction ID 的 S5F4 回覆。
2. **檢查 null** — `reply` 為 `null` 表示 T3 timeout 或設備無回應。
3. **`reply.SecsItem.FirstValue<byte>()`** — 根 Item 直接是 `B:1`，取第一個 byte 即為 ACKC5。
4. **判斷結果** — `ACKC5 == 0` 表示成功。

```csharp
// 在 OutboundCommands.cs 的 SendS5F3EnableAllAlarmsAsync 內，解析 S5F4
SecsMessage? reply = await secsGem.SendAsync(s5f3, ct);

if (reply?.SecsItem is null)
{
    Console.WriteLine("[WARN] S5F3R: no reply (T3 timeout or null body)");
    return;
}

// S5F4 格式：B:1 ACKC5（根節點直接是 B:1，無 List 包裝）
byte ackc5 = reply.SecsItem.FirstValue<byte>();
if (ackc5 != 0)
    Console.WriteLine($"[WARN] S5F4 ACKC5={ackc5} — alarm enable/disable rejected by equipment");
else
    Console.WriteLine("[RX] S5F4 ACKC5=0 — alarm enable/disable confirmed");
```

> **為何不需要 `IMessageHandler`：** secs4net v2 中，凡是 Host 主動發出並等待回覆的訊息（primary with reply bit），回覆由 `SendAsync` 直接返回，不會進入 `MessageDispatcher` 的處理迴圈。只有設備**主動發起**的 primary message（如 S5F1R、S6F11）才需要 `IMessageHandler`。

### 完整的 S5F3R → S5F4 流程

S5F4 的解析在 `OutboundCommands.cs` 的 S5F3R 發送方法中完成，請參閱 [S5F3R.md](S5F3R.md) 的「C# 發送程式碼（secs4net v2）」章節取得完整實作。

## 舊版 API 程式碼（secs4net v1 / SecsPort）

以下 C# 程式碼使用舊版 `secs4net`（`SecsPort` 時代）API，展示 **Host 端** 發送 S5F3 後解析 S5F4 回覆的邏輯。**不適用於本專案**，僅供對照參考。

### 處理流程

1. **發送 S5F3 並等待回覆** — 呼叫 `sp.SendSecsMsg(5, 3, true, sendData, true)` 發送 enable/disable alarm 請求，其中 `true` 表示需要等待回覆。
2. **檢查回傳結果** — `rcr.rc != 0` 表示通訊失敗（例如 timeout），直接跳出。
3. **解析 ACKC5** — 用 `SecsPort.ListSplit()` 拆解回覆資料，驗證 type tag 為 `B:1`，再用 `sp.BinToInt()` 將 binary 轉為整數。
4. **判斷結果** — `ACKC5 == 0` 表示成功啟用/停用。

```csharp
    // S5F4 C# Parse reply
    SecsHost sp=(SecsHost)your_SecsHost;
    RcResult rcr = sp.SendSecsMsg(5, 3, true, sendData, true);
    string TSN_data;
    bool ok=true;
    while (ok) {   // break out of loop on error
        if (rcr.rc != 0) {ok=false; break;}
        TSN_data = rcr.result;
        // expect ACKC5
        // variables for data items and parsing
        int ACKC5;  	// B:1 (always)  acknowledge code, 0 ok
        string [] list0;
        list0 = SecsPort.ListSplit(TSN_data);
        if (list0.Length < 1) { ok = false; break; }
        if (list0[0] != "B:1") { ok = false; break; }
        ACKC5 = sp.BinToInt(list0[1]);
        return;  // parsed ok
        } // end while(ok)
```

## 實測 Log 案例（KA01 Schmoll 鑽孔機，2026-03-18）

以下為 S5F3R / S5F4 的實際 XML log，可對照確認設備回覆結構：

```xml
<!-- Host 發送 S5F3R：ALED=128（Enable），ALID 空值（廣播） -->
<SECSMessage s="5" f="3" direction="H to E" replyBit="true"
             txid="2000035" time="2026-03-18T06:09:00.493Z" deviceID="0">
    <SECSData>
        <LST>
            <BIN>128</BIN>  <!-- ALED=0x80 = Enable -->
            <UI4 />         <!-- ALID 空值 = 套用至全部 alarm -->
        </LST>
    </SECSData>
</SECSMessage>

<!-- 設備回覆 S5F4：ACKC5=0（OK），根節點直接是 BIN，無 LST 包裝 -->
<SECSMessage s="5" f="4" direction="E to H" replyBit="false"
             txid="2000035" time="2026-03-18T06:09:00.495Z" deviceID="0">
    <SECSData>
        <BIN>0</BIN>  <!-- ACKC5=0 = OK -->
    </SECSData>
</SECSMessage>
```

**觀察重點：**

| 欄位 | 值 | 說明 |
|------|-----|------|
| `replyBit` (S5F3R) | `true` | Host 要求設備回覆 |
| `replyBit` (S5F4) | `false` | 回覆訊息不需再回覆 |
| S5F4 根結構 | `<BIN>0</BIN>` | 直接是 `B:1`，無 `<LST>` 包裝 |
| ACKC5 | `0` | 設備確認全部警報已啟用 |
| 來回時間 | 2 ms（493Z → 495Z）| 設備回覆迅速 |
