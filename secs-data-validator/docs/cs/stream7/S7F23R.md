# S7F23R - Formatted Process Program Send

**Direction:** Sent by Host and Equipment

---

| **S7F23R** | Formatted Process Program Send | Sent by Host and Equipment |

### 用意與目的

S7F23R（**Formatted Process Program Send**）是 SECS-II Stream 7（製程程式管理）的核心訊息，用於 **Host 主動將製程配方（recipe）上傳至設備**，或設備向 Host 回報自身當前配方。

#### 在 GEM 工作流程中的定位

```
Host                              Equipment
  │                                   │
  │── S7F23R (上傳配方 RECIPE_HOT) ──▶│  ← 本訊息
  │◀─────────── S7F24 (ACKC7=0x00) ──│  設備確認儲存
  │                                   │
  │── S2F41R (START RECIPE_HOT) ─────▶│  之後才執行
  │◀─────────── S2F42 (HCACK=0x00) ──│
```

S7F23R 屬於 **SEMI GEM（E30）規範的必要能力之一**（Required GEM Capability #7：Process Program Management）。設備必須實作此訊息的接收與驗證。

#### 訊息名稱解析

| 部分 | 含義 |
|------|------|
| `S7` | Stream 7 — 製程程式管理 |
| `F23` | Function 23 — 第 23 號功能（Send） |
| `R` | Reply requested（需要回覆）— 發送方期待收到 S7F24 作為 ACK |

> **為何需要回覆（R）？** 配方上傳是有狀態的操作：設備可能儲存空間不足（`0x04`）、配方已存在（`0x03`）或正在使用中（`0x06`）。`R` 確保 Host 得到明確的接受/拒絕結果，而非靜默失敗。

#### 與相關訊息的關係

| 訊息 | 方向 | 用途 |
|------|------|------|
| **S7F23R** | Host → Equipment | 上傳格式化配方（本文件） |
| S7F24 | Equipment → Host | 確認/拒絕接收（ACKC7） |
| S7F5R | Host → Equipment | 請求設備回傳配方（下載） |
| S7F6 | Equipment → Host | 設備回傳配方內容 |
| S7F17R | Host → Equipment | 刪除指定配方 |
| S7F19R | Host → Equipment | 查詢設備上的配方清單 |

### Format (TSN)

```
{L:4
  [PPID]
  [MDLN]
  [SOFTREV]
  {L:c
    {L:2
      [CCODE]
      {L:p
        [PPARM]
      }
    }
  }
}
```

**Format Explanation:**

這是一個**四層巢狀結構**，用來傳送帶有格式化命令和參數的製程程式：

- **`{L:4}`** — 外層 list，固定包含 4 個元素：
  1. **`[PPID]`** — Process Program ID（製程程式 ID），型別 `A:80`（ASCII 字串，最大 80 字元）。識別要傳送的製程程式名稱。允許空字串（代表預設程式）。
  2. **`[MDLN]`** — Equipment Model Type（設備型號），型別 `A:20`（固定 20 字元）。指定此製程程式適用的設備型號。**必須為 ASCII 型別，否則視為格式錯誤。**
  3. **`[SOFTREV]`** — Software Revision（軟體版本），型別 `A:20`（固定 20 字元）。指定此製程程式適用的軟體版本。**必須為 ASCII 型別，否則視為格式錯誤。**
  4. **`{L:c}`** — 命令清單 list，包含 `c` 個命令（`c` 為變數，可為 0），每個命令是一個 `{L:2}` 結構：
     - **`[CCODE]`** — Process Operation Command Code（製程操作命令碼），型別 `A:n`（可變長度 ASCII 字串）。定義一個操作步驟或命令。
     - **`{L:p}`** — 該命令的參數 list，包含 `p` 個參數（`p` 為變數，可為 0）：
       - **`[PPARM]`** — Process Parameter（製程參數），型別 `A:n`（可變長度）。可以是任意純量或向量型別的 SECS 資料項目（ASCII、U4、F4 等）。

### CCODE 深度解析

#### CCODE 是什麼？

**CCODE（Process Operation Command Code，製程操作命令碼）** 是格式化製程程式（Formatted Process Program）中最核心的設計概念。它的型別為 `A:n`（可變長度 ASCII 字串），在 S7F23R 的巢狀結構中扮演「**指令名稱**」的角色 — 每個 CCODE 代表一個製程操作步驟，搭配零或多個 PPARM（參數），構成一條完整的命令。

用程式設計的比喻：**CCODE 是函式名稱，PPARM 是引數列表**。

```
函式呼叫：  TEMP_SET(300.0)
SECS 結構：  {L:2  A "TEMP_SET"  {L:1  F4 300.0}}
              └─ CCODE ──┘       └── PPARM ──┘
```

#### 為什麼需要 CCODE？（Formatted vs. Unformatted）

SEMI E5 標準定義了兩種製程程式傳輸方式，CCODE 是兩者之間的根本差異：

| 特性 | **Formatted**（格式化，含 CCODE） | **Unformatted**（非格式化，純文字） |
|------|----------------------------------|--------------------------------------|
| 訊息 | S7F23R / S7F25R / S7F26 | S7F3R / S7F4 / S7F5R / S7F6 |
| 內容結構 | 巢狀 `L:4 { PPID, MDLN, SOFTREV, {L:c {L:2 CCODE {L:p PPARM}}}}` | 單一 `A:n` 或 `B:n` 位元組流 |
| Host 能否解析 | ✅ **語意級別解析** — 每個命令和參數有明確型別與意義 | ❌ **不透明** — Host 只看到一坨文字/二進位，無法理解內部結構 |
| Host 能否驗證 | ✅ 可逐一檢查 CCODE 是否合法、參數是否在合理範圍內 | ❌ 只能做整體 byte-by-byte 比對 |
| Host 能否修改 | ✅ 可修改單一 PPARM（例如微調溫度參數）而不需重傳整份配方 | ❌ 必須由設備端或人工離線編輯 |
| 跨設備可攜性 | ✅ MDLN + SOFTREV 指定適用機型，避免不相容配方被載入錯誤設備 | ❌ 設備型號資訊不包含在訊息結構中 |

**CCODE 的設計用意**正是解決非格式化配方的三大痛點：

1. **不透明（Opaque）：** 非格式化配方對 Host 來說是一個黑盒子。Host 無法知道配方裡設定了多少溫度、壓力參數，只能原封不動地儲存與轉發。CCODE 讓每個操作步驟變成**具名、有型別的結構化資料**，Host 可以讀取、解析、甚至修改單一參數。

2. **不可驗證（Non-verifiable）：** 非格式化配方無法在通訊層驗證語意正確性。CCODE 搭配 S7F22（Process Capabilities Data）提供的元資料，使接收方能驗證：
   - 命令是否為設備支援的合法 CCODE
   - 參數數量與型別是否符合設備定義
   - 命令之間的**順序約束**（例如「加熱」必須在「通氣」之前）

3. **不可攜（Non-portable）：** 非格式化配方缺乏設備識別資訊。CCODE 所在的 `L:4` 結構包含 `MDLN` 和 `SOFTREV`，讓接收方可以在接收時就判斷配方是否適用於自己的機型與軟體版本，避免不相容配方被錯誤載入。

#### CCODE 與 S7F22（Process Capabilities）的關係

S7F22（Process Capabilities Data）是 S7F21 的回覆，它為每個 CCODE 提供完整的元資料定義。這些元資料是**設備自我描述**能力的體現：

```
S7F22 中的每個 CCODE 定義（L:11 結構）：
┌──────────────────────────────────────────────────────┐
│ CCODE   A:n   — 命令碼識別符                           │
│ CNAME   A:16  — 人類可讀名稱（例如 "Temperature Set"）  │
│ RQCMD   TF:1  — 此命令是否為必要（每個配方都必須包含）    │
│ BLKDEF  I1:1  — 命令區塊定義關係                       │
│ BCDS    U2:n  — 必須在此命令之前出現的 CCODE 列表        │
│ IBCDS   U2:n  — 必須緊接在此命令之前的 CCODE 列表       │
│ NBCDS   U2:n  — 不可在此命令之前出現的 CCODE 列表       │
│ ACDS    U2:1  — 必須在此命令之後出現的 CCODE            │
│ IACDS   U2:n  — 必須緊接在此命令之後的 CCODE 列表       │
│ NACDS   U2:n  — 不可在此命令之後出現的 CCODE 列表       │
│ {L:p}         — 參數定義區塊                           │
└──────────────────────────────────────────────────────┘
```

**順序約束的實際意義：**

BCDS/IBCDS/NBCDS/ACDS/IACDS/NACDS 這六個欄位定義了 CCODE 之間的**先後順序規則**。例如一個蝕刻設備可能定義：

- `TEMP_SET` 的 BCDS 包含 `CHAMBER_PURGE` → 溫度設定必須在腔體吹淨之後
- `RF_ON` 的 NBCDS 包含 `GAS_FLOW` → RF 開啟前不可先通入製程氣體
- `VENT` 的 IBCDS 包含 `RF_OFF` → 腔體洩壓前必須緊接著關閉 RF

這些約束讓設備在收到 S7F23R 時，能**自動驗證配方邏輯正確性**，而非盲目接受任何命令序列。

#### CCODE 的使用規則

1. **CCODE 是設備廠商定義的：** SEMI E5 只定義了傳輸格式（`A:n` + `{L:p PPARM}`），CCODE 的具體名稱和語意由設備廠商在其 Equipment Capability Specification（ECS）中定義。例如同一個「設定溫度」的操作，A 廠商可能叫 `TEMP_SET`，B 廠商可能叫 `TC_SET_VALUE`。

2. **CCODE 的 PPARM 型別必須一致：** 同一個 CCODE 在整條產線的 PPARM 型別應保持一致。混用型別（同一 CCODE 今天傳 F4、明天傳 U4）會造成設備端解析失敗。

3. **命令清單是有序的：** `{L:c}` 中的 CCODE 順序即為執行順序。設備應按照列表中的順序依序執行。

4. **允許重複 CCODE：** 同一個 CCODE 可以在命令清單中出現多次（例如多步製程中每步都有溫度設定），每次可帶不同 PPARM 值。

5. **空命令清單（c = 0）是合法的：** 代表一個只識別設備但無實際操作步驟的配方，Handler 會接受並回傳 `ACKC7 = 0x00`。

#### CCODE 的實際範例

以下是一個半導體蝕刻製程的 CCODE 命令序列，展示格式化配方如何將製程步驟表達為結構化指令：

```
L:4
  A "ETCH_RECIPE_001"       ← PPID（配方名稱）
  A "EtchSystem-A"          ← MDLN（適用設備型號）
  A "v2.1.0"                ← SOFTREV（適用軟體版本）
  L:6                       ← 6 個 CCODE 命令
    L:2  A "CHAMBER_PURGE"   L:1  A "N2"          ← 1. 用氮氣吹淨腔體
    L:2  A "TEMP_SET"        L:1  F4 250.0        ← 2. 升溫至 250°C
    L:2  A "PRESS_SET"       L:1  F4 3.0          ← 3. 設定壓力 3 Torr
    L:2  A "GAS_FLOW"        L:2  A "CF4" F4 50.0 ← 4. 通入 CF4 氣體 50 sccm
    L:2  A "RF_ON"           L:2  U4 300 F4 13.56 ← 5. 開啟 RF 300W / 13.56MHz
    L:2  A "ETCH_TIME"       L:1  U4 60           ← 6. 蝕刻 60 秒
```

對應的 C# 建構：
```csharp
var recipe = L(
    A("ETCH_RECIPE_001"),
    A("EtchSystem-A"),
    A("v2.1.0"),
    L(
        L(A("CHAMBER_PURGE"), L(A("N2"))),
        L(A("TEMP_SET"),      L(F4(250.0f))),
        L(A("PRESS_SET"),     L(F4(3.0f))),
        L(A("GAS_FLOW"),      L(A("CF4"), F4(50.0f))),
        L(A("RF_ON"),         L(U4(300u), F4(13.56f))),
        L(A("ETCH_TIME"),     L(U4(60u)))
    )
);
```

> **對比：** 若使用非格式化配方（S7F3R），同樣的製程程式只是一個 `A:512` 或更大的 ASCII 字串，Host 無法解析其中的溫度、壓力值，也無法驗證 `RF_ON` 是否在 `GAS_FLOW` 之後。這正是 CCODE 存在的根本價值。

### 資料型別說明與範例

S7F23R 訊息中使用到的 SECS-II 資料型別如下。型別分為兩層規則：**固定 ASCII**（PPID、MDLN、SOFTREV、CCODE）和**彈性多型**（PPARM）。

#### 型別速查表

| 欄位 | SECS 型別 | secs4net 工廠方法 | C# 對應型別 | 強制性 |
|------|-----------|-----------------|-------------|--------|
| PPID | `A:n`（最大 80 字元） | `A("RECIPE_001")` | `string` | 允許空字串 |
| MDLN | `A:20` | `A("EtchSystem-A")` | `string` | **必須 ASCII** |
| SOFTREV | `A:20` | `A("v1.0.0")` | `string` | **必須 ASCII** |
| CCODE | `A:n` | `A("TEMP_SET")` | `string` | 建議 ASCII |
| PPARM | 任意 SECS 型別 | `F4(300.0f)` / `U4(5u)` / `A("LOW")` / … | 視型別而定 | 無限制 |

---

#### `A` — ASCII 字串

**用途：** PPID、MDLN、SOFTREV、CCODE 全部使用此型別。`A` 是 SECS-II 中最常見的純文字型別。

```csharp
// secs4net 建構
Item ppid    = A("RECIPE_001");   // A:10
Item mdln    = A("EtchSystem-A"); // A:12
Item softrev = A("v1.0.0");       // A:6
Item ccode   = A("TEMP_SET");     // A:8

// 讀取
string value = item.GetString();  // item.Count > 0 時使用
```

SML 表示：
```
A "RECIPE_001"
```

> **注意：** 若 MDLN 或 SOFTREV 不是 `A` 型別（例如誤傳 `U4`），Handler 會立即回傳 `ACKC7 = 0x02`（Length error）。

---

#### `F4` — 32 位元浮點數（IEEE 754 Single）

**用途：** PPARM 最常見的數值型別。適合溫度（°C）、壓力（Torr）、流量（sccm）等連續量測值。

```csharp
// secs4net 建構
Item temp  = F4(300.0f);  // F4:1 → 300.0
Item press = F4(5.0f);    // F4:1 → 5.0

// 讀取
float val = item.FirstValue<float>();
```

SML 表示：
```
F4 300.0
F4 5.0
```

完整 S7F23 命令範例（溫度設定 300°C）：
```csharp
L(A("TEMP_SET"), L(F4(300.0f)))
// → {L:2  A "TEMP_SET"  {L:1 F4 300.0}}
```

---

#### `F8` — 64 位元浮點數（IEEE 754 Double）

**用途：** 需要高精度的 PPARM，例如奈米級定位座標。

```csharp
Item pos = F8(0.00012345);

float val = item.FirstValue<double>();
```

SML 表示：
```
F8 0.00012345
```

---

#### `U4` — 無號 32 位元整數

**用途：** 計數值、步驟編號、電源設定值等非負整數參數。

```csharp
Item step  = U4(3u);      // 第 3 步
Item power = U4(1500u);   // 1500 W

uint val = item.FirstValue<uint>();
```

SML 表示：
```
U4 3
U4 1500
```

---

#### `U2` — 無號 16 位元整數

**用途：** 範圍在 0–65535 的計數器或小型設定值（節省傳輸位元組）。

```csharp
Item rpm = U2(3600);

ushort val = item.FirstValue<ushort>();
```

SML 表示：
```
U2 3600
```

---

#### `U1` — 無號 8 位元整數

**用途：** 旗標值、枚舉代碼、比例因子（0–255）。

```csharp
Item flag = U1(1);

byte val = item.FirstValue<byte>();
```

SML 表示：
```
U1 1
```

---

#### `I4` — 有號 32 位元整數

**用途：** 可為負數的參數，例如偏移量（offset）或相對位移。

```csharp
Item offset = I4(-50);    // 偏移 -50

int val = item.FirstValue<int>();
```

SML 表示：
```
I4 -50
```

---

#### `BOOLEAN` — 布林值

**用途：** 開關旗標，例如「是否啟用冷卻」、「是否跳過清洗步驟」。

```csharp
Item enable = Boolean(true);

bool val = item.FirstValue<bool>();
```

SML 表示：
```
BOOLEAN True
```

---

#### `B` — 二進位（Binary Byte Array）

**用途：** ACKC7 回覆碼使用 `B:1`；也可用於原始韌體指令或自訂 bitmap。

```csharp
// S7F24 回覆用
Item ack = B(0x00);       // Accepted

// 多位元組 bitmap
Item bitmap = B(0xFF, 0x0A, 0x03);

byte val = item.FirstValue<byte>();
```

SML 表示：
```
B 0x00
```

---

#### PPARM 型別選擇指引

實際應用中，選擇哪種 PPARM 型別取決於**物理量的性質**和**精度需求**：

| 場景 | 建議型別 | 原因 |
|------|----------|------|
| 溫度（°C）、壓力（Torr）、流量（sccm） | `F4` | 工業製程精度 ±0.001 已足夠，F4 節省 4 位元組 |
| 奈米級座標、光學精度量測 | `F8` | F4 精度不足（約 7 位有效數字），需 F8（15 位） |
| 步驟編號、計數器、電源（W） | `U4` | 非負整數，上限 ~42 億，最通用 |
| 轉速（rpm）、小型計數器（0–65535） | `U2` | 節省 2 位元組，適合範圍確定的小整數 |
| 旗標、枚舉代碼（0–255） | `U1` | 最省空間，適合有限值域的離散量 |
| 偏移量（可負值）、相對位移 | `I4` | 唯一能表示負整數的常用型別 |
| 開關控制（啟用/停用） | `BOOLEAN` | 語意明確，避免用 `U1(1)` 代替 |
| 氣體名稱、模式字串 | `A` | 人類可讀的列舉值，例如 `"N2"`、`"LOW"` |
| 韌體 bitmap、原始指令 | `B` | 需要逐 bit 控制時 |

> **設計原則：** 同一個 CCODE 的 PPARM 型別在整條產線應保持一致。混用型別（今天 F4、明天 U4）會造成設備端解析失敗。建議在設備規格書（ECS）中明確記載每個 CCODE 的 PPARM 型別。

#### 多型 PPARM 組合範例

實際製程程式常混用多種型別的 PPARM，以下展示一個含 5 種型別的完整命令清單：

```csharp
// 包含 5 種 PPARM 型別的製程程式
var root = L(
    A("RECIPE_MULTI"),            // PPID
    A("EtchSystem-A"),            // MDLN
    A("v2.0.0"),                  // SOFTREV
    L(
        // 命令 1：溫度設定（F4）
        L(A("TEMP_SET"),  L(F4(300.0f))),

        // 命令 2：壓力設定（F4）
        L(A("PRESS_SET"), L(F4(5.0f))),

        // 命令 3：步驟編號（U4）
        L(A("STEP_NO"),   L(U4(3u))),

        // 命令 4：啟用冷卻（BOOLEAN）
        L(A("COOLING"),   L(Boolean(true))),

        // 命令 5：氣體名稱（A）
        L(A("GAS_ID"),    L(A("N2"))),

        // 命令 6：多參數（F4 + U2）— 一個命令可帶多個 PPARM
        L(A("RAMP_CTRL"), L(F4(10.0f), U2(60)))
    )
);
```

對應的 SML 表示：
```
L:4
  A "RECIPE_MULTI"
```

### 回覆訊息：S7F24

收到 S7F23R 後，接收方必須回覆 **S7F24**，內容為一個 `B:1` 的 ACKC7 回覆碼：

| ACKC7 值 | 說明 |
|----------|------|
| `0x00` | Accepted（已接受） |
| `0x01` | Command not supported（不支援此命令） |
| `0x02` | Length error（格式或長度錯誤） |
| `0x03` | Already exists（製程程式已存在） |
| `0x04` | Insufficient space（儲存空間不足） |
| `0x06` | In use（製程程式使用中） |

### 通訊流程

```
Host / Equipment              接收方
       |── S7F23R ──────────────▶|   傳送格式化製程程式
       |                          |   解析並驗證內容
       |◀─────────── S7F24 ───────|   回覆 ACKC7 (B:1)
```

### C# secs4net 實作

以下實作使用本專案的 `IMessageHandler` 策略介面和 `secs4net` 函式庫。

#### 檔案位置

- **Handler：** `SecsApp/Handlers/S7F23Handler.cs`
- **測試：** `SecsApp.Tests/Handlers/S7F23HandlerTests.cs`
- **註冊：** `SecsApp/Program.cs`（`factory.Register(new S7F23Handler())`）

#### 處理流程

1. **驗證外層 L:4** — 確認 root 是 `SecsFormat.List` 且 `Count == 4`，否則回傳 `ACKC7 = 0x02`。
2. **驗證 MDLN / SOFTREV 型別** — 必須是 `SecsFormat.ASCII`，否則拒絕。
3. **讀取 PPID / MDLN / SOFTREV** — PPID 允許空字串；MDLN 和 SOFTREV 用 `Item.GetString()` 讀取。
4. **驗證命令清單** — `root[3]` 必須是 `SecsFormat.List`。
5. **迴圈解析每個命令** — 每個命令必須是 `L:2`；第二個元素（參數清單）必須是 `SecsFormat.List`。
6. **PPARM 型別彈性** — 參數可為任意 SECS 型別，用 `Item.ToString()` 轉字串記錄，不強制 ASCII。
7. **回覆 S7F24** — 解析全部成功後回傳 `ACKC7 = 0x00`；任何驗證失敗回傳 `0x02`。

#### S7F23Handler.cs

```csharp
// SecsApp/Handlers/S7F23Handler.cs
using Secs4Net;
using static Secs4Net.Item;

namespace SecsApp.Handlers;

public sealed class S7F23Handler : IMessageHandler
{
    public int Stream   => 7;
    public int Function => 23;

    // ACKC7 values per SEMI E5
    public const byte Ackc7Accepted          = 0x00;
    public const byte Ackc7CmdNotSupported   = 0x01;
    public const byte Ackc7LengthError       = 0x02;
    public const byte Ackc7AlreadyExists     = 0x03;
    public const byte Ackc7InsufficientSpace = 0x04;
    public const byte Ackc7InUse            = 0x06;

    public async Task HandleAsync(PrimaryMessageWrapper wrapper, CancellationToken ct)
    {
        byte ackc7 = Parse(wrapper.PrimaryMessage.SecsItem);

        using var s7f24 = new SecsMessage(7, 24)
        {
            Name     = "FormattedProcessProgramAck",
            SecsItem = B(ackc7),
        };
        bool sent = await wrapper.TryReplyAsync(s7f24, ct);
        Console.WriteLine($"[TX] S7F24 ACKC7=0x{ackc7:X2} → {(sent ? "OK" : "Already replied")}");
    }

    // internal 以便單元測試可存取（透過 InternalsVisibleTo）
    internal static byte Parse(Item? root)
    {
        if (root is null || root.Format != SecsFormat.List || root.Count != 4)
        {
            Console.WriteLine("[WARN] S7F23: unexpected root structure — rejecting");
            return Ackc7LengthError;
        }

        if (root[1].Format != SecsFormat.ASCII)
        {
            Console.WriteLine("[WARN] S7F23: MDLN is not ASCII — rejecting");
            return Ackc7LengthError;
        }

        if (root[2].Format != SecsFormat.ASCII)
        {
            Console.WriteLine("[WARN] S7F23: SOFTREV is not ASCII — rejecting");
            return Ackc7LengthError;
        }

        string ppid    = ReadAscii(root[0]);
        string mdln    = ReadAscii(root[1]);
        string softrev = ReadAscii(root[2]);
        Console.WriteLine($"[RX] S7F23 PPID=\"{ppid}\" MDLN=\"{mdln}\" SOFTREV=\"{softrev}\"");

        var commandList = root[3];
        if (commandList.Format != SecsFormat.List)
        {
            Console.WriteLine("[WARN] S7F23: command list is not a List — rejecting");
            return Ackc7LengthError;
        }

        for (int c = 0; c < commandList.Count; c++)
        {
            var cmdItem = commandList[c];
            if (cmdItem.Format != SecsFormat.List || cmdItem.Count != 2)
            {
                Console.WriteLine($"[WARN] S7F23: command[{c}] is not L:2 — rejecting");
                return Ackc7LengthError;
            }

            var paramList = cmdItem[1];
            if (paramList.Format != SecsFormat.List)
            {
                Console.WriteLine($"[WARN] S7F23: command[{c}] param list is not a List — rejecting");
                return Ackc7LengthError;
            }

            string ccode = ReadAscii(cmdItem[0]);
            var parms = Enumerable.Range(0, paramList.Count)
                .Select(p => paramList[p].ToString() ?? string.Empty)
                .ToList();

            Console.WriteLine($"  CMD[{c}] CCODE=\"{ccode}\" PPARM=[{string.Join(", ", parms)}]");
        }

        return Ackc7Accepted;
    }

    private static string ReadAscii(Item item) =>
        item.Format == SecsFormat.ASCII && item.Count > 0
            ? item.GetString()
            : string.Empty;
}
```

### Host 端發送 S7F23R（OutboundCommands.cs）

`SendS7F23AndWaitS7F24Async()` 封裝了 Host 主動上傳製程程式的完整流程：建構 S7F23R、透過 HSMS 發送、等待並解析 S7F24 回覆。

#### 檔案位置

- **實作：** `SecsApp/OutboundCommands.cs`

#### 方法簽章

```csharp
internal static async Task<byte> SendS7F23AndWaitS7F24Async(
    ISecsGem secsGem,
    string ppid,
    string mdln,
    string softrev,
    IEnumerable<(string ccode, IEnumerable<Item> pparms)> commands,
    CancellationToken ct)
```

| 參數 | 型別 | 說明 |
|------|------|------|
| `ppid` | `string` | 製程程式 ID（A:80，允許空字串） |
| `mdln` | `string` | 設備型號（A:20） |
| `softrev` | `string` | 軟體版本（A:20） |
| `commands` | `IEnumerable<(string, IEnumerable<Item>)>` | 命令清單，PPARM 可為任意 secs4net `Item` 型別（`F4`、`U4`、`A` 等） |
| 回傳 | `Task<byte>` | ACKC7 byte（`0x00` = 成功） |

#### 實作（摘自 OutboundCommands.cs）

```csharp
// SecsApp/OutboundCommands.cs
var cmdItems = commands
    .Select(cmd => L(A(cmd.ccode), L(cmd.pparms.ToArray())))
    .ToArray();

using var s7f23 = new SecsMessage(7, 23)
{
    Name     = "FormattedProcessProgramSend",
    SecsItem = L(A(ppid), A(mdln), A(softrev), L(cmdItems)),
};

Console.WriteLine($"[TX] S7F23R PPID=\"{ppid}\" MDLN=\"{mdln}\" SOFTREV=\"{softrev}\" commands={cmdItems.Length}");
using var s7f24 = await secsGem.SendAsync(s7f23, ct);

var root = s7f24.SecsItem;
if (root is null || root.Format != SecsFormat.Binary || root.Count != 1)
{
    Console.WriteLine("[ERROR] S7F24: expected ACKC7 (B:1)");
    return S7F23Handler.Ackc7LengthError;
}

byte ackc7 = root.FirstValue<byte>();
Console.WriteLine($"[INFO] S7F24 ACKC7=0x{ackc7:X2} ({Ackc7Description(ackc7)})");
return ackc7;
```

### 單元測試

#### 測試策略說明

`PrimaryMessageWrapper` 是 secs4net 的 `sealed` 類別，建構子為 `internal`，無法在測試中直接建立或繼承。
解法：將解析邏輯提取為 `internal static Parse(Item? root)`，讓測試可直接呼叫，不需要 TCP 連線或 secs4net 內部狀態機。

透過在 `SecsApp.csproj` 加入以下設定，讓測試專案可存取 `internal` 成員：

```xml
<AssemblyAttribute Include="System.Runtime.CompilerServices.InternalsVisibleTo">
  <_Parameter1>SecsApp.Tests</_Parameter1>
</AssemblyAttribute>
```
