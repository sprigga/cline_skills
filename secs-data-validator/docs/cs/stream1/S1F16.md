# S1F16 - OFF-LINE Acknowledge

**Direction:** Sent by Equipment Only

---

| **S1F16** | OFF-LINE Acknowledge | Sent by Equipment Only |

## Format

**Data Item:** [OFLACK](http://hume.com/secs/items.html#OFLACK) — Binary (B:1) value containing the offline acknowledge status.

- **0** = Success (equipment accepted the offline request)
- **Non-zero** = Failure (equipment rejected or encountered an error)

## C# Implementation Example

This code demonstrates how to parse an S1F16 reply in C# using the `secs4net` library:

```csharp
// S1F16 C# Parse reply
SecsHost sp = (SecsHost)your_SecsHost;
RcResult rcr = sp.SendSecsMsg(1, 15, true, sendData, true);
string TSN_data;
bool ok = true;

while (ok) {   // break out of loop on error
    if (rcr.rc != 0) { ok = false; break; }
    TSN_data = rcr.result;
    
    // Expect OFLACK (Binary:1)
    int OFLACK;
    string[] list0;
    
    // Split the reply data structure
    list0 = SecsPort.ListSplit(TSN_data);
    if (list0.Length < 1) { ok = false; break; }
    
    // Verify data type is Binary:1
    if (list0[0] != "B:1") { ok = false; break; }
    
    // Extract the acknowledge value (0 = success)
    OFLACK = sp.BinToInt(list0[1]);
    return;  // parsed ok
} // end while(ok)
```

**Code breakdown:**
- `SendSecsMsg(1, 15, ...)` — Sends S1F15 (offline request) and waits for S1F16 reply
- `ListSplit()` — Parses the reply TSN data into list elements
- `BinToInt()` — Converts the binary data to an integer value
- **0 = Success**, non-zero indicates an error condition

## 實戰案例：OFLACK=0 的語義與 S6F11R 時序

### 場景

在以下真實案例中，Equipment 回傳 S1F16（OFLACK=0）後，在 10ms 內送出了 S6F11R (CEID=2 EquipmentOffline)：

```
09:36:22.553  Host      → S1F15R  (W=1)
09:36:22.572  Equipment → S1F16  BIN=0  (OFLACK=0)
09:36:22.582  Equipment → S6F11R  CEID=2             ← 此順序正確（見下方說明）
09:36:22.595  Host      → S6F12  BIN=0  (ACK)
```

### OFLACK 值的含義

| OFLACK 值 | 說明 | Host 後續動作 |
|-----------|------|--------------|
| `0` | 接受，Equipment 已完成狀態轉換，進入 OFF-LINE | 預期收到 CEID=2 的 S6F11R 事件通知 |
| 非零 | 拒絕，Equipment 維持 ON-LINE | 可重試或通知操作員 |

### S6F11R 在 S1F16 之後的時序說明

OFLACK=0 的語義是「**我已**完成狀態轉換，進入 OFF-LINE」。Collection Event CEID=2 (`EquipmentOffline`) 是在狀態轉換**完成後**才觸發，因此 S6F11R **必然**在 S1F16 之後送出，**此順序符合 GEM 規範**，不是問題。

> **注意：** 此案例早期曾誤判 S6F11R 在 S1F16 之後出現為「race condition」，經重新審視後確認為正確行為。真正的 issue 是 RPTID=2 原始定義缺少 VID 2004（已修正），詳見 `S6F11R.md`。

### Host 端解析 S1F16 的正確做法（C#）

```csharp
// Host 端解析 S1F16 後
RcResult rcr = sp.SendSecsMsg(1, 15, true, sendData, true);
bool ok = true;
while (ok) {
    if (rcr.rc != 0) { ok = false; break; }
    string TSN_data = rcr.result;
    string[] list0 = SecsPort.ListSplit(TSN_data);
    if (list0.Length < 1) { ok = false; break; }
    if (list0[0] != "B:1") { ok = false; break; }
    int OFLACK = sp.BinToInt(list0[1]);

    if (OFLACK == 0) {
        // Equipment 已進入 OFF-LINE
        // 預期接下來會收到 S6F11R (CEID=2 EquipmentOffline)，應正常 ACK
        SetGemState(HostState.EquipmentOffline);
    }
    return;
} // end while(ok)
```
