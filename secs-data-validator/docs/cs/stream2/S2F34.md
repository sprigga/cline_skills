**Direction:** Equipment Only

| **S2F34** | Define Report Acknowledge | Equipment Only |

Format:

[DRACK](http://hume.com/secs/items.html#DRACK)

        if (list0[0] != "B:1") { ok = false; break; }
        DRACK = sp.BinToInt(list0[1]);
        return;  // parsed ok
        } // end while(ok)

## Understanding Format and Codebase

### Format
The **Format** section shows a simple scalar reply:
- `[DRACK]` — a single binary item (B:1) representing the define report acknowledgment code

### Codebase (C# / secs4net)
This is a **reply parsing example** on the Host side (receiving S2F34 after sending S2F33):
1. **Send request and wait**: `sp.SendSecsMsg(2, 33, true, sendData, true)` sends S2F33 and blocks for the reply
2. **Check transmission status**: `rcr.rc != 0` indicates a communication failure (e.g., timeout, device offline)
3. **Parse successful reply**:
   - `SecsPort.ListSplit(TSN_data)` extracts the message
   - Validate that `list0[0] == "B:1"` (binary type, 1 byte)
   - `sp.BinToInt(list0[1])` converts the binary byte to an integer (0 or 1 typically)
4. **Extract and use**: Store DRACK in a variable for application logic

**Key difference from S2F31R/S2F32**:
- S2F31R is an **Equipment receiver** (passively waits for Host to send)
- S2F32 and S2F34 are **Host receivers** (actively send and wait for replies)
- The parsing logic is the same; the difference is in the protocol flow and who initiates.
