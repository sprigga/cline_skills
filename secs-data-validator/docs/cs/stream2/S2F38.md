# S2F38 - Enable/Disable Event Report Acknowledge

**Direction:** Equipment Only

---

| **S2F38** | Enable/Disable Event Report Acknowledge | Equipment Only |

Format:

[ERACK](http://hume.com/secs/items.html#ERACK)

```
    // S2F38 C# Parse reply
    SecsHost sp=(SecsHost)your_SecsHost;
    RcResult rcr = sp.SendSecsMsg(2, 37, true, sendData, true);
    string TSN_data;
    bool ok=true;
    while (ok) {   // break out of loop on error
        if (rcr.rc != 0) {ok=false; break;}
        TSN_data = rcr.result;
        // expect ERACK
        // variables for data items and parsing
        int ERACK;  	// B:1 (always)  enable/disable event report acknowledge
        string [] list0;
        list0 = SecsPort.ListSplit(TSN_data);
        if (list0.Length < 1) { ok = false; break; }
        if (list0[0] != "B:1") { ok = false; break; }
        ERACK = sp.BinToInt(list0[1]);
        return;  // parsed ok
        } // end while(ok)
```

## Understanding Format and Codebase

### Format
The **Format** section shows a simple scalar reply:
- `[ERACK]` — a single binary item (B:1) representing the enable/disable event report acknowledgment code

### ERACK Values

| ERACK | Meaning |
|-------|---------|
| **0** | **OK — request accepted** |
| 1 | Denied, insufficient space |
| 2 | Denied, invalid format |
| 3 | Denied, CEID does not exist |
| 4 | Denied, RPTID does not exist |
| 5 | Denied, VID does not exist |

**`ERACK = 0` is the only success value.** Any non-zero value means the equipment rejected the enable/disable request.

### Codebase (C# / secs4net)
This is a **Host-side reply parsing example** (receiving S2F38 after sending S2F37):
1. **Send request and wait**: `sp.SendSecsMsg(2, 37, true, sendData, true)` sends S2F37 and blocks for the reply
2. **Check transmission**: `rcr.rc != 0` indicates failure at the transport layer (e.g., T3 timeout — no reply received)
3. **Parse successful reply**:
   - `SecsPort.ListSplit(TSN_data)` extracts the message
   - Validate type: `list0[0] == "B:1"`
   - `sp.BinToInt(list0[1])` converts binary to integer
4. **Validate the acknowledgment**: Check `ERACK == 0` for application-layer success

### Two-Layer Success Check

A complete success check requires passing **two gates**:

```csharp
// Gate 1 — transport layer: reply was received within T3 timeout
if (rcr.rc != 0) { ok = false; break; }

// ... parse ERACK ...
ERACK = sp.BinToInt(list0[1]);

// Gate 2 — application layer: equipment accepted the request
if (ERACK != 0)
{
    Console.WriteLine($"S2F38 ERACK={ERACK}: enable/disable events rejected");
    ok = false; break;
}
// ERACK == 0 → success, equipment has enabled/disabled the specified event reports
```

- `rcr.rc != 0` — HSMS-level failure; no S2F38 was received at all
- `ERACK != 0` — S2F38 was received, but the equipment denied the request

This two-gate pattern is identical to how `OutboundCommands.cs` validates `HCACK` in `SendS2F41AndWaitS2F42Async()`.

### ERACK = 1: Denied, Insufficient Space

ERACK=1 means the equipment lacks memory or internal resources to accommodate the event report configuration implied by the enable/disable request.

This almost always occurs when the host has not cleared stale report definitions from a previous connection before reconfiguring. The full GEM initialization sequence is:

```
S2F33 (empty list) → clear all RPTID definitions
S2F35             → link CEID → RPTID
S2F37             → enable/disable event reports
S2F38             ← ERACK=0 (success)
```

If S2F33 is skipped and old definitions accumulate across connections, S2F37 may trigger ERACK=1.

**Common root causes:**

| Cause | Explanation |
|-------|-------------|
| Too many active RPTIDs | Equipment has a hard limit on simultaneously defined reports; prior S2F33 calls filled it |
| VID list too large | A single RPTID references too many Variable IDs for the equipment to cache on enable |
| Too many RPTIDs per CEID | S2F35 linked one CEID to too many reports; buffer overflow on trigger |
| Stale definitions not cleared | Previous connection left report definitions in equipment memory |

**Resolution:** Send S2F33 with an empty list (`DATAID=0, L:0`) to wipe all RPTID definitions before re-linking with S2F35 and re-enabling with S2F37.

### Acknowledgment Pattern

All Stream 2 acknowledge messages (S2F32, S2F34, S2F36, S2F38) follow the identical structure:
- Simple binary scalar (B:1) response
- Same parsing: type validation → `BinToInt` → value check against `0`
