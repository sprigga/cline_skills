# S2F31R - Date and Time Set Request

**Direction:** Host Only

---

| **S2F31R** | Date and Time Set Request | Host Only |

Comment: Modern equipment should use NTP servers to synchronize the system clock. SECS is inaccurate and does not include timezone info.

Format:

[TIME](http://hume.com/secs/items.html#TIME)

## Understanding Format and Codebase

### Format
The **Format** section shows the SECS-II message structure using SECS notation:
- `[TIME]` — a single item representing the time value, encoded as ASCII (A:32) per the ECV TimeFormat control

The item is wrapped in SECS encoding, so the actual wire format encodes type information (e.g., `A:12` for 12-byte ASCII).

### Codebase (C# / secs4net)
The C# code example shows how to:
1. **Register** the handler: `sp.MessageTypeAdd(2, 31, ...)` tells the SecsPort to route S2F31 messages to `recv_S2F31R`
2. **Parse** the incoming data:
   - `SecsPort.ListSplit(TSN_data)` breaks the message into elements
   - `list0[0]` should equal `"A:32"` (the type code for TIME)
   - `list0[1]` contains the actual time string
3. **Validate**: Check that the structure matches expectations (type code, length)
4. **Extract**: Pull the TIME value into a string variable
5. **Send reply**: If `send_reply` is true, use `sp.SendReply(2, 32, transID, reply)` to send the acknowledgment (S2F32)
6. **Error handling**: On any parse error, call `sp.SendS9(7, header)` to send an "illegal data" message

**Key difference**: The Format describes the logical structure; the Codebase shows how the secs4net library represents that structure as a parsed string with type codes.

## Observed Log Example (KA01 Schmoll S50, 2025-12-17)

```xml
<SECSMessage s="2" f="31" direction="H to E" replyBit="true" txid="2000027"
             time="2025-12-17T07:12:12.475Z" deviceID="0">
  <SECSData><ASC>2025121715062050</ASC></SECSData>
</SECSMessage>
<SECSMessage s="2" f="32" direction="E to H" replyBit="false" txid="2000027"
             time="2025-12-17T07:12:12.485Z" deviceID="0">
  <SECSData><BIN>0</BIN></SECSData>
</SECSMessage>
```

### Analysis

| Field | Value | Notes |
|-------|-------|-------|
| TIME (S2F31) | `2025121715062050` | A:16, TimeFormat=1: `YYYYMMDDHHMMSS` + centiseconds `50` → 2025-12-17 15:06:20.50 (CST) |
| UTC offset | UTC+8 | Log timestamp is `07:12:12Z`; TIME value is `15:06:20 CST` — 8h ahead confirms Asia/Shanghai TZ |
| TIACK (S2F32) | `BIN:0` | 0 = Accepted. Equipment acknowledged the time set without error. |
| Round-trip latency | ~10 ms | `475Z` → `485Z` |

### Key Observations

- Equipment uses **TimeFormat=1** (16-char `YYYYMMDDHHMMSScc`), not the 12-char format.
- TIME value is **local time (CST/UTC+8)**, not UTC — consistent with the ECV TimeFormat convention that excludes timezone info.
- `S2F32` body is a bare `BIN:0` scalar (TIACK), **not** wrapped in a list — parse with `SecsPort.ListSplit()` directly on the BIN token.
- Result: ✅ **OK** — equipment accepted the Host time synchronization.
