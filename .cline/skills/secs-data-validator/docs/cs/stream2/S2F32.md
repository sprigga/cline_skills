# S2F32 - Date and Time Set Acknowledge

**Direction:** Equipment → Host (reply only, no Reply bit)

---

| **S2F32** | Date and Time Set Acknowledge | Equipment Only |

## Purpose and Intent

S2F32 is the **acknowledgment reply** sent by the Equipment in response to an S2F31R (Date and Time Set Request) from the Host. It completes the **time synchronization handshake** defined in SEMI E5 (SECS-II), Stream 2 (Equipment Control and Diagnostics).

The sole purpose of this message is to confirm whether the Equipment accepted or rejected the time set operation. It carries a single data item — `TIACK` — whose binary value indicates the outcome.

### Role in the S2F31R / S2F32 Round-Trip

```
Host                              Equipment
  │                                   │
  │──── S2F31R (TIME = "20251217...") ─▶│   Host sets the Equipment clock
  │                                   │   Equipment validates and applies
  │◀─── S2F32 (TIACK = 0x00) ─────────│   Equipment confirms acceptance
  │                                   │
```

- **S2F31R** is Host-initiated, carries the new time value (ASCII, A:12 or A:16)
- **S2F32** is Equipment-originated, carries only the `TIACK` result byte
- No further messages follow — TIACK is the final word on the operation

### When Is S2F32 Used?

S2F32 is emitted whenever the Equipment receives a valid S2F31R. The Host sends S2F31R to:
- Synchronize the Equipment clock at startup (before running jobs)
- Periodically correct clock drift without NTP
- Compensate for environments where NTP is unavailable or untrusted

> **Note:** Modern equipment should prefer NTP. SECS time sync is limited: it has
> ~10 ms round-trip latency and carries **no timezone information**. The time value
> is always local time per ECV `TimeFormat`, not UTC.

## Message Format

```
S2F32 body:
  TIACK   B:1   (always present, never empty)
```

`TIACK` is a **bare scalar** — it is the direct message body, not wrapped in a `List`. This is a common SECS-II pattern for simple acknowledge messages.

### TIACK Values

| Value (hex) | Value (dec) | Meaning |
|-------------|-------------|---------|
| `0x00` | 0 | **Accepted** — time set succeeded |
| `0x01` | 1 | **Error** — time set rejected (format invalid, out of range, etc.) |

Any value other than `0x00` should be treated as a rejection. The SEMI E5 standard defines only 0 and 1; vendor extensions are non-standard.

---

## Data Item Detail: TIACK

| Attribute | Value |
|-----------|-------|
| Type code | `B` (Binary) |
| Length | 1 byte always |
| secs4net format | `SecsFormat.Binary` |
| Read with | `root.FirstValue<byte>()` |
| Valid values | `0x00` = OK, `0x01` = Error |

Binary (`B`) in SECS-II means an opaque byte sequence, **not** a boolean or a numeric integer. The value is interpreted contextually by each message definition. For TIACK, the byte is treated as an unsigned code.

## C# Implementation (secs4net)

### Host Side — Parsing the S2F32 Reply

When the Host sends S2F31R and awaits S2F32, secs4net returns the reply as a `SecsMessage`. The Host parses `TIACK` from the reply body:

```csharp
// Host side: send S2F31R and wait for S2F32
internal static async Task<bool> SendS2F31AndWaitS2F32Async(
    ISecsGem secsGem,
    DateTime time,
    CancellationToken ct)
{
    // Build TIME string in TimeFormat=1: YYYYMMDDHHMMSScc
    string timeStr = time.ToString("yyyyMMddHHmmss") + "00"; // cc = centiseconds

    using var s2f31 = new SecsMessage(2, 31)
    {
        Name     = "DateTimeSet",
        SecsItem = Item.A(timeStr),   // A:16 ASCII time string
    };

    Console.WriteLine($"[TX] S2F31R TIME={timeStr}");
    using var s2f32 = await secsGem.SendAsync(s2f31, ct);
    Console.WriteLine($"[RX] S2F32 {s2f32.Name}: {s2f32.SecsItem}");

    // S2F32 body: TIACK is a bare B:1 scalar (no List wrapper)
    var root = s2f32.SecsItem;
    if (root is null || root.Format != SecsFormat.Binary || root.Count != 1)
    {
        Console.WriteLine("[ERROR] S2F32: expected TIACK (B:1)");
        return false;
    }

    byte tiack = root.FirstValue<byte>();
    Console.WriteLine($"[INFO] S2F32 TIACK=0x{tiack:X2} ({TiackDescription(tiack)})");
    return tiack == 0x00;
}

private static string TiackDescription(byte tiack) => tiack switch
{
    0x00 => "Accepted",
    0x01 => "Error — time set rejected",
    _    => $"Unknown (0x{tiack:X2})",
};
```

### Equipment Side — Sending S2F32 (Handler)

If this codebase were acting as Equipment (passive mode), the S2F31R handler would reply with S2F32:

```csharp
// Equipment side: S2F31R handler that sends S2F32
public sealed class S2F31Handler : IMessageHandler
{
    public int Stream   => 2;
    public int Function => 31;

    public async ValueTask HandleAsync(PrimaryMessageWrapper wrapper)
    {
        var msg  = wrapper.PrimaryMessage;
        var body = msg.SecsItem;

        // Validate: TIME must be A:12 or A:16 ASCII
        if (body is null || body.Format != SecsFormat.ASCII)
        {
            // Malformed — reply with TIACK = 0x01 (error)
            await wrapper.TryReplyAsync(new SecsMessage(2, 32)
            {
                Name     = "DateTimeSetAck",
                SecsItem = Item.B(0x01),  // TIACK = Error
            });
            return;
        }

        string time = body.GetString();
        Console.WriteLine($"[RX] S2F31R TIME={time}");

        // Apply time (platform-specific; simplified here)
        bool applied = TryApplyTime(time);
        byte tiack   = applied ? (byte)0x00 : (byte)0x01;

        // Reply with S2F32 TIACK — bare B:1 scalar, no list wrapper
        await wrapper.TryReplyAsync(new SecsMessage(2, 32)
        {
            Name     = "DateTimeSetAck",
            SecsItem = Item.B(tiack),
        });
    }

    private static bool TryApplyTime(string time)
    {
        // A:16 format: YYYYMMDDHHMMSScc  (cc = centiseconds, ignored for system clock)
        // A:12 format: YYMMDDHHmmSS
        if (time.Length == 16 &&
            DateTime.TryParseExact(time[..14], "yyyyMMddHHmmss",
                null, System.Globalization.DateTimeStyles.None, out var dt))
        {
            // Would call SetSystemTime() here on real equipment
            Console.WriteLine($"[INFO] Applying time: {dt:O}");
            return true;
        }
        Console.WriteLine($"[WARN] Cannot parse TIME value: {time}");
        return false;
    }
}
```

## Data Format Examples

### Wire Format (Binary / HSMS)

On the wire, S2F32 encodes as:

```
12-byte HSMS header  +  3 bytes body
                         └─ [format=0x21 (B), length=1]  [value=0x00]
```

The body encoding `0x21 0x01 0x00` means:
- `0x21` = Binary format (upper nibble `0x2_`) + 1-byte length field
- `0x01` = 1 byte of data follows
- `0x00` = TIACK value (Accepted)

### secs4net Item Representation

```csharp
// Create TIACK = 0 (Accepted)
Item tiackOk    = Item.B(0x00);   // SecsFormat.Binary, Count=1

// Create TIACK = 1 (Error)
Item tiackError = Item.B(0x01);   // SecsFormat.Binary, Count=1

// Read the value back
byte tiack = tiackOk.FirstValue<byte>();   // → 0
int  tiackInt = tiack;                     // safe widening; use byte internally

// Type-safe check (avoid magic numbers)
bool accepted = tiack == 0x00;
```

In the secs4net log, S2F32 appears as:

```xml
<SECSMessage s="2" f="31" direction="H to E" replyBit="true" txid="2000027"
             time="2025-12-17T07:12:12.475Z" deviceID="0">
  <SECSData><ASC>2025121715062050</ASC></SECSData>
</SECSMessage>
<SECSMessage s="2" f="32" direction="E to H" replyBit="false" txid="2000027"
             time="2025-12-17T07:12:12.485Z" deviceID="0">
  <SECSData><BIN>0</BIN></SECSData>
</SECSMessage>
```

### Analysis

| Field | Value | Notes |
|-------|-------|-------|
| TIME (S2F31) | `2025121715062050` | A:16, TimeFormat=1: `YYYYMMDDHHMMSS` + centiseconds `50` → 2025-12-17 15:06:20.50 (CST) |
| UTC offset | UTC+8 | Log timestamp is `07:12:12Z`; TIME value is `15:06:20 CST` — 8 h ahead confirms Asia/Shanghai TZ |
| TIACK (S2F32) | `BIN:0` | 0x00 = Accepted — Equipment acknowledged the time set without error |
| Round-trip latency | ~10 ms | `475Z` → `485Z` |

### Key Observations

- Equipment uses **TimeFormat=1** (16-char `YYYYMMDDHHMMSScc`), not the 12-char format.
- TIME value is **local time (CST/UTC+8)**, not UTC — consistent with the ECV TimeFormat convention that carries no timezone info.
- `S2F32` body is a **bare `BIN:0` scalar**, not wrapped in a List — parse directly via `root.FirstValue<byte>()`.
- Result: ✅ **OK** — equipment accepted the Host time synchronization.

## Common Mistakes

| Mistake | Correct Approach |
|---------|-----------------|
| Expecting `L:1 { B:1 }` (List wrapper) | TIACK is a bare `B:1` — check `root.Format == SecsFormat.Binary` directly |
| Using `(int)tiack != 0` | Use `tiack == 0x00` or compare the byte value directly |
| Sending UTC time in S2F31R | Send **local time** unless the Equipment's ECV TimeFormat explicitly includes timezone |
| Ignoring TIACK = 1 | Always check the reply; proceed only when TIACK = 0x00 |

## Relationship to Other Messages

| Message | Direction | Role |
|---------|-----------|------|
| **S2F31R** | Host → Equipment | Carries `TIME` (A:12 or A:16); initiates clock sync |
| **S2F32** | Equipment → Host | Carries `TIACK` (B:1); acknowledges the request |
| S2F17R | Host → Equipment | Request current time from Equipment (reverse direction) |
| S2F18 | Equipment → Host | Equipment replies with current TIME value |
