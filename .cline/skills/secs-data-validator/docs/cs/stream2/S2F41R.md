## Understanding Format and Codebase

### Format
The **Format** section shows a two-level structure:
```
{L:2
  [RCMD]
  {L:n
    {L:2
      [CPNAME]
      [CPVAL]
    }
  }
}
```
- **Outer list (L:2)**: Contains RCMD (remote command name) and a list of command parameters
- **Parameter list (L:n)**: Variable-length list of parameter name/value pairs, each as an L:2 sublist

### Codebase (C# / secs4net)
The C# code implements two-level nested parsing with an inner L:2 structure:
1. **Level 0**: Split the outer L:2 list
2. **Extract RCMD**: Get the command name from `list0[1]`
3. **Process parameter list**:
   - `list1 = SecsPort.ListSplit(list0[2])` parses the variable-length parameter list
   - Loop through `list1[1..n]` with counter `n`
4. **Level 2 (each parameter pair)**:
   - For each parameter, `list2 = SecsPort.ListSplit(list1[n])` parses the inner L:2 structure
   - Validate: `list2[0] == "L:2"`
   - Extract CPNAME from `list2[1]`
   - Extract CPVAL from `list2[2]`
5. **Reply**: Send S2F42 acknowledgment
6. **Error handling**: Send S9F7 on parse failure

**Key semantic**:
- RCMD must be a printable string (hex 21-7E, max 20 characters per GEM spec)
- Parameters are arbitrary name/value pairs
- CPVAL can be any scalar type (the equipment must know the expected type for each CPNAME)

**Difference from S2F33R/S2F35R**:
- S2F33R and S2F35R have **two separate variable-length lists** (L:a and L:b)
- S2F41R has **one variable-length list containing L:2 pairs** (the parameter pairs are fixed L:2 structures, not recursive lists)
