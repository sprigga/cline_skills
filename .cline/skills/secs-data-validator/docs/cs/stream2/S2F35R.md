**Direction:** Host Only

---

[RPTID](http://hume.com/secs/items.html#RPTID)

 }

} }

## 訊息目的與用意

### S2F35R 在 GEM 事件報告系統中的角色

S2F35R（Link Event Report）是 GEM 標準事件報告配置流程的**第二步**，負責建立「**收集事件（CEID）→ 報告（RPTID）**」的對應關係。

GEM 事件報告系統的完整初始化需要三個訊息依序執行：

```
步驟 1: S2F33R  Define Report      → 定義報告內容（RPTID 包含哪些 VID）
步驟 2: S2F35R  Link Event Report  → 連結事件與報告（CEID 觸發哪些 RPTID）
步驟 3: S2F37R  Enable Event Report → 啟用事件（哪些 CEID 開始主動回報）
```

**S2F35R 的核心語義**：「當設備發生事件 CEID-X 時，請同時回傳 RPTID-1、RPTID-2、…等報告的資料。」

### 訊息方向

`Host Only` — 由 Host（主機）發出，設備（Equipment）接收並回覆 S2F36（LRACK 確認碼）。

### 重要語義注意事項

1. **不保證輸出順序**：此訊息連結的是「哪些報告應隨事件一起發送」，但不定義報告資料的傳輸順序（設備實作自由決定）。
2. **b=0 的特殊意義**：當 RPTID 清單長度為 0 時，表示**解除該 CEID 與所有報告的連結**（disassociate），等同於清除此 CEID 的所有報告設定。
3. **多對多關係**：一個 CEID 可以連結多個 RPTID；不同 CEID 也可以共用相同的 RPTID。

## 資料格式詳細說明

### 巢狀結構圖解

```
{L:2
  [DATAID]        ← 交易識別碼（U4:1 或其他 Un 型態）
  {L:a            ← a 個事件連結定義（a=0 表示清除所有連結）
    {L:2          ← 每個事件連結為一個 L:2 結構
      [CEID]      ← 收集事件 ID（U4:1，GEM 要求 Un 型態）
      {L:b        ← b 個報告 ID（b=0 表示解除此 CEID 的所有報告連結）
        [RPTID]   ← 報告 ID（U4:1，GEM 要求 Un 型態）
      }
    }
  }
}
```

### 資料項目型態說明

| 資料項目 | SECS 型態 | C# 對應型態 | 說明 |
|---------|-----------|-------------|------|
| **DATAID** | `U4:1`（可變，Un 均可） | `string`（解析後轉 uint） | 用於關聯同一批配置訊息的識別碼，通常遞增計數 |
| **CEID** | `U4:1`（GEM 要求 Un） | `string`（解析後轉 uint） | 收集事件識別碼，由設備廠商定義 |
| **RPTID** | `U4:1`（GEM 要求 Un） | `string`（解析後轉 uint） | 報告識別碼，由 S2F33R 事先定義 |

> **型態注意**：GEM 標準要求 CEID 和 RPTID 使用無號整數（Un），但 SECS-II 允許其他型態（如 ASCII 字串 `A:n`）。廠商的設備通常使用 `U4:1`，因此程式碼中以 `string` 暫存後再轉換。

## 完整資料範例

### 範例一：單一事件連結兩個報告

**情境**：當「晶圓處理完成事件」（CEID=100）發生時，同時回傳「製程報告」（RPTID=1）和「計量報告」（RPTID=2）。

```
S2F35R W
<L:2
  <U4:1 1>          // DATAID = 1（第一批配置訊息）
  <L:1              // a = 1 個事件連結
    <L:2
      <U4:1 100>    // CEID = 100（UnitProcessed，晶圓處理完成）
      <L:2          // b = 2 個報告
        <U4:1 1>    // RPTID = 1（製程報告，由 S2F33R 定義）
        <U4:1 2>    // RPTID = 2（計量報告，由 S2F33R 定義）
      >
    >
  >
>
```

設備回覆：
```
S2F36
<B:1 0x00>          // LRACK = 0（成功）
```

### 範例二：多個事件各自連結不同報告

**情境**：配置兩個不同事件，各自觸發不同報告。

```
S2F35R W
<L:2
  <U4:1 2>          // DATAID = 2
  <L:2              // a = 2 個事件連結
    <L:2
      <U4:1 100>    // CEID = 100（UnitProcessed）
      <L:1          // b = 1 個報告
        <U4:1 1>    // RPTID = 1（製程報告）
      >
    >
    <L:2
      <U4:1 1001>   // CEID = 1001（ProductionState，狀態變更）
      <L:2          // b = 2 個報告
        <U4:1 3>    // RPTID = 3（狀態報告）
        <U4:1 4>    // RPTID = 4（診斷報告）
      >
    >
  >
>
```

### 範例三：b=0 解除 CEID 的所有報告連結

**情境**：清除 CEID=100 與所有報告的關聯（disassociate）。

```
S2F35R W
<L:2
  <U4:1 3>          // DATAID = 3
  <L:1              // a = 1 個事件連結
    <L:2
      <U4:1 100>    // CEID = 100
      <L:0>         // b = 0 → 解除此 CEID 的所有報告連結
    >
  >
>
```

### 範例四：a=0 清除所有事件連結

**情境**：在重新配置前，先清除設備上所有的事件報告連結。

```
S2F35R W
<L:2
  <U4:1 4>          // DATAID = 4
  <L:0>             // a = 0 → 無任何事件連結（清除全部）
>
```

> **GEM 初始化慣例**：Host 通常在連線後先以 `a=0` 的 S2F35R 清除所有連結，再重新以 `a>0` 建立所需的 CEID→RPTID 對應，確保設備狀態乾淨。

## 在 secs4net 架構中的實作說明

### 格式解析層次

程式碼遵循三層巢狀解析模式（與 S2F33R 完全相同的結構）：

```
list0 = ListSplit(TSN_data)   → 解析最外層 L:2（DATAID + 事件清單）
list1 = ListSplit(list0[1])   → 解析 DATAID 的 item（U4:1）
list1 = ListSplit(list0[2])   → 解析 L:a 事件清單
  loop a: list2 = ListSplit(list1[a])   → 解析每個 L:2（CEID + RPTID 清單）
    list3 = ListSplit(list2[1])         → 解析 CEID
    list3 = ListSplit(list2[2])         → 解析 L:b RPTID 清單
      loop b: list4 = ListSplit(list3[b]) → 解析每個 RPTID
```

### 索引對齊規則

`SecsPort.ListSplit()` 的回傳陣列中：
- `[0]` — 型態碼（如 `"L:2"`、`"U4:1"`）
- `[1..n]` — 實際元素值

因此判斷 `list0.Length != 3` 代表「型態碼 + 2 個元素」，對應 `L:2` 結構。

### 驗證邏輯

每層解析都有嚴格的驗證：
- 外層 `L:2` 必須有 `list0[0] == "L:2"` 且 `list0.Length == 3`
- 每個事件的內層 `L:2` 同樣驗證 `list2[0] == "L:2"` 且 `list2.Length == 3`
- 任一層驗證失敗，`ok = false` 跳出迴圈並回傳 S9F7（非法資料）

### 回覆與錯誤處理

| 回覆 | 條件 |
|------|------|
| `S2F36（LRACK）` | 解析成功，`send_reply == true` 時發送 |
| `S9F7` | 任何層次的格式驗證失敗 |

## 與相關訊息的關係

### GEM 事件報告配置流程

```
Host → Equipment : S2F33R  定義報告（RPTID 包含哪些 VID）
Host ← Equipment : S2F34   確認（DRACK）

Host → Equipment : S2F35R  連結事件（CEID → RPTID）
Host ← Equipment : S2F36   確認（LRACK）

Host → Equipment : S2F37R  啟用事件（CEED + CEID 清單）
Host ← Equipment : S2F38   確認（ERACK）

-- 事件發生時 --
Equipment → Host : S6F11R  事件報告（CEID + 對應的 RPTID 資料）
Host ← Equipment : S6F12   確認
```

### 三個訊息的職責分工

| 訊息 | 職責 | 類比說明 |
|------|------|---------|
| **S2F33R** | 定義報告模板（RPTID → VID 清單） | 決定「報告要包含哪些欄位」 |
| **S2F35R** | 連結事件與報告（CEID → RPTID 清單） | 決定「事件發生時要觸發哪些報告」 |
| **S2F37R** | 啟用/停用事件（CEED + CEID 清單） | 決定「哪些事件被監聽」 |

### LRACK 回覆碼

S2F36 的 LRACK（Link Report Acknowledge）值：

| LRACK 值 | 含義 |
|---------|------|
| `0x00` | 成功（Acknowledge, command performed） |
| `0x01` | 不支援（Denied, at least one CEID link already defined, clear event report first） |
| `0x02` | 無法執行（Denied, at least one CEID does not exist） |
| `0x03` | 資料超出範圍（Denied, at least one RPTID does not exist） |

> **注意**：執行 S2F35R 前，通常需確保 RPTID 已由 S2F33R 事先定義，否則設備可能回覆 LRACK=3（RPTID 不存在）。

## 與 S2F33R 的結構比較

S2F35R 和 S2F33R 使用完全相同的三層巢狀結構，差別僅在語義層次：

| 比較項 | S2F33R（Define Report） | S2F35R（Link Event Report） |
|--------|------------------------|---------------------------|
| **外層 L:2** | DATAID + 報告清單 | DATAID + 事件連結清單 |
| **L:a 中每個元素** | 一個報告定義（RPTID + VID 清單） | 一個事件連結（CEID + RPTID 清單） |
| **L:b 內容** | VID（變數 ID，字串型） | RPTID（報告 ID，數值型） |
| **a=0 語義** | 刪除所有報告及事件連結 | 清除所有事件報告連結 |
| **b=0 語義** | 刪除此 RPTID 及其事件連結 | 解除此 CEID 與所有報告的連結 |
| **回覆訊息** | S2F34（DRACK） | S2F36（LRACK） |
| **C# 解析演算法** | 完全相同 | 完全相同 |
