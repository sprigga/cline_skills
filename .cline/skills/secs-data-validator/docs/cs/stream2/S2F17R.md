# S2F17R - Date and Time Request

**Direction:** Host and Equipment

---

| **S2F17R** | Date and Time Request | Host and Equipment |

Comment: NTP servers should be used instead for better accuracy and to avoid lack of time zone specification.

## Format

`_header only_` — This message contains only the SECS-II message header with **no data items**. It is a simple request that requires no parameters. The corresponding reply **S2F18** (Date and Time Data) carries the actual date/time information.

## C# Code Reference (secs4net)

The code below demonstrates how to register and handle S2F17R using the `secs4net` C# SECS-II library:

- **`sp.MessageTypeAdd(2, 17, ...)`** — Registers the receive handler `recv_S2F17R` on a `SecsPort` or `SecsHost` object so it is called whenever an S2F17R message arrives.
- **`recv_S2F17R(sender, stream, function, send_reply, transID, TSN_data, header)`** — The handler callback. Parameters:
  - `sender` — The `SecsPort` or `SecsHost` that received the message.
  - `stream`, `function` — Message stream/function (2, 17).
  - `send_reply` — Whether the caller expects a reply (`true` → send S2F18).
  - `transID` — Transaction ID used to correlate request and reply.
  - `TSN_data` — TSN-encoded data payload (empty for this header-only message).
  - `header` — Raw message header string.
- **`sp.SendReply(2, 18, transID, reply)`** — Sends S2F18 reply. The `reply` content is TBD (to be implemented with the actual date/time data).

## Observed Log Example (KA01 Schmoll S50, 2025-12-17)

```xml
<SECSMessage s="2" f="17" direction="H to E" replyBit="true" txid="2000029"
             time="2025-12-17T07:13:15.324Z" deviceID="0">
  <SECSData />
</SECSMessage>
<SECSMessage s="2" f="18" direction="E to H" replyBit="false" txid="2000029"
             time="2025-12-17T07:13:15.336Z" deviceID="0">
  <SECSData><ASC>2025121715072335</ASC></SECSData>
</SECSMessage>
```

### Analysis

| Field | Value | Notes |
|-------|-------|-------|
| S2F17 body | *(empty)* | Header-only message — no data items, confirming the format spec |
| TIME (S2F18) | `2025121715072335` | A:16, TimeFormat=1: `YYYYMMDDHHMMSS` + centiseconds `35` → 2025-12-17 15:07:23.35 (CST) |
| UTC offset | UTC+8 | Log timestamp is `07:13:15Z`; TIME value is `15:07:23 CST` — 8h ahead confirms Asia/Shanghai TZ |
| Round-trip latency | ~12 ms | `324Z` → `336Z` |

### Key Observations

- S2F17 has **no data body** (`<SECSData />`); the reply S2F18 carries all the time information.
- Equipment returns time in **TimeFormat=1** (16-char `YYYYMMDDHHMMSScc`) — consistent with the S2F32 reply observed ~63 seconds earlier in the same session.
- TIME value is **local time (CST/UTC+8)**, not UTC — same convention as S2F31.
- This S2F17/S2F18 exchange occurred ~63 s after S2F31/S2F32 (`07:12:12Z` → `07:13:15Z`), suggesting the Host set the time and then immediately queried it back to verify synchronization.
- Result: ✅ **OK** — equipment returned current local time without error.
