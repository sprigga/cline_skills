# S6F11R - Event Report Send

**Direction:** Sent by Equipment Only

---

## 用意和目的 (Purpose and Intent)

### S6F11R 是什麼？

S6F11R 是 GEM (SEMI E30) 標準中**最重要的事件通知訊息**。設備透過此訊息向主機 (Host) **主動回報**狀態變更、警報觸發、製程完成等事件。它是整個 GEM 資料收集子系統的核心——所有非輪詢式 (non-polling) 的狀態通知都依賴 S6F11R。

### 為什麼需要 S6F11R？

在半導體/面板製造環境中，Host 需要即時掌握設備狀態，但**持續輪詢** (polling) 會浪費通訊頻寬且延遲高。S6F11R 採用 **事件驅動 (event-driven)** 模型：

| 方式 | 行為 | 效率 |
|------|------|------|
| 輪詢 (S1F3/F4) | Host 定期問「你現在什麼狀態？」 | 低效，有延遲 |
| **S6F11R 事件** | 設備狀態變更時**主動推送** | 高效，零延遲 |

### S6F11R 在 GEM 標準中的定位

S6F11R 屬於 **Collection Event Management**（集合事件管理）框架的一部分，與以下訊息協同運作：

```
                        GEM Collection Event Framework
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  S2F33/F34  ── 定義報告模板 (Define Report)                 │
│       ↓                                                     │
│  S2F35/F36  ── 將報告模板綁定到 CEID (Link Event↔Report)    │
│       ↓                                                     │
│  S2F37/F38  ── 啟用/停用特定 CEID (Enable/Disable Event)    │
│       ↓                                                     │
│  ★ S6F11/F12 ── 設備觸發事件時自動發送 (Event Report) ★     │
│                                                             │
│  S6F15/F16  ── Host 主動查詢事件報告 (Event Report Request) │
│  S6F19/F20  ── Host 查詢單一報告模板內容 (Individual Report)│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**關鍵設計理念：** S6F11R 的資料內容**不是硬編碼**的。Host 先透過 S2F33/S2F35/S2F37 設定「哪些事件要回報」以及「每個事件要包含哪些變數」，然後設備才會在事件觸發時按照設定組裝 S6F11R 的內容。這就是 **CEID → RPTID → VID** 三層間接定址架構。

### 核心概念記憶法

> **CEID = 「什麼事發生了？」** (Which event occurred?)
> **RPTID = 「要回報哪些數據？」** (Which data to report?)
> **VID = 「每個數據的語義？」** (What does each value mean?)
> **V = 「數據的實際值」** (The actual value)

## Format

**Message Structure:** Equipment sends an event report triggered by a collection event.

### Format 規範

S6F11R 採用**三層巢狀串列 (three-level nested list)** 結構：

```
{L:3
[DATAID]
[CEID]
{L:a
{L:2
[RPTID]
{L:b
[V]
}
}
}}
```

### 結構逐層解析

| 層級 | 元素 | SECS 型別 | 說明 |
|------|------|-----------|------|
| **第 0 層** | `{L:3 ...}` | List | 最外層串列，固定包含 3 個子元素 |
| 第 0 層 [0] | `[DATAID]` | `U4:1` | Data ID — 訊息關聯識別碼，用於多區塊傳輸或多訊息關聯 |
| 第 0 層 [1] | `[CEID]` | `U4:1` | Collection Event ID — 觸發此報告的事件識別碼 (GEM 規範要求型別為 Un) |
| 第 0 層 [2] | `{L:a ...}` | List | 變長串列，包含 1 到多個報告 (Report)，`a` 為報告數量 |
| **第 1 層** | `{L:2 ...}` | List | 每份報告固定 2 個子元素 |
| 第 1 層 [0] | `[RPTID]` | `U4:1` | Report ID — 指向 S2F33 定義的報告模板 |
| 第 1 層 [1] | `{L:b ...}` | List | 變長串列，包含此報告模板的所有變數值，`b` 為 VID 數量 |
| **第 2 層** | `[V]` | Any | Variable Value — 可以是任何 SECS 型別 (U1, U4, A, B, L, ...) |

### 圖解結構（以實際資料為例）

以下以 CEID=2 (EquipmentOffline) 為例，展示完整的三層結構：

```
S6F11R
├── DATAID = 4                    ← U4 (4-byte unsigned int)
├── CEID   = 2                    ← U4 (Collection Event ID)
└── Reports (L:a = 1 份報告)
    └── Report #0
        ├── RPTID = 2             ← U4 (Report Template ID)
        └── Values (L:b = 7 個變數)
            ├── V[0] = "2026041417355095"  ← ASC (設備時鐘 VID:2004)
            ├── V[1] = 3                    ← UI1 (控制狀態 VID:2028)
            ├── V[2] = 0                    ← UI4 (控制開關 VID:2033)
            ├── V[3] = 1                    ← UI4 (線上開關 VID:2034)
            ├── V[4] = 256                  ← UI4 (通訊開關 VID:2035)
            ├── V[5] = 260                  ← UI4 (通訊狀態 VID:2036)
            └── V[6] = 4                    ← UI1 (前控制狀態 VID:4030)
```

### 為什麼是三層結構？

三層設計提供了**極大的彈性**：

1. **同一事件可攜帶多份報告** (第 1 層迴圈 `a`)
   - 例如：CEID=9 (ProcessingCompleted) 可能同時攜帶「製程結果報告」和「設備狀態報告」兩份 RPTID

2. **同一報告可攜帶多個變數** (第 2 層迴圈 `b`)
   - 例如：RPTID=2 包含 7 個 VID（時鐘、控制狀態、通訊狀態等）

3. **變數型別不限** (V 可以是任何 SECS 型別)
   - 簡單值：`U4`, `I1`, `A`（字串）
   - 複雜值：巢狀 `L`（如 CEID=1004 的警報列表）

## 資料格式與型態詳解 (Data Format and Type Details)

### SECS-II 型別對照表

S6F11R 中出現的所有 SECS 資料型別，以及對應的 secs4net C# API：

| SECS 型別 | 說明 | 線上標記 | secs4net 建立 | secs4net 解析 | 出現位置 |
|-----------|------|---------|--------------|-------------|---------|
| **U4** | 4-byte 無號整數 | `<UI4>` | `Item.U4(4)` | `.FirstValue<uint>()` | DATAID, CEID, RPTID |
| **U1** | 1-byte 無號整數 | `<UI1>` | `Item.U1(3)` | `.FirstValue<byte>()` | 部分 VID 值 (如 CONTROLSTATE) |
| **A** | ASCII 字串 | `<ASC>` | `Item.A("text")` | `.GetString()` | 部分 VID 值 (如 Clock, ProcessState) |
| **B** | Binary (原始位元組) | `<BIN>` | `Item.B(0x00)` | `.FirstValue<byte>()` | S6F12 回覆的 ACKC6 |
| **L** | List (串列) | `<LST>` | `Item.L(item1, item2, ...)` | `.Count`, `[]` 索引 | 結構容器 |

### 型別使用範例

#### 1. U4 — DATAID, CEID, RPTID

```
線上格式：<UI4>2</UI4>
```

```csharp
// secs4net 建立
var ceid = Item.U4(2);

// secs4net 解析（從 S6F11R 訊息中取出 CEID）
ulong ceidValue = root[1].FirstValue<uint>();  // root[1] 是 CEID 元素
Console.WriteLine($"CEID = {ceidValue}");       // 輸出：CEID = 2
```

> **注意：** `FirstValue<uint>()` 回傳 `uint`，但我們賦值給 `ulong`，因為 CEID 在本專案中以 `const ulong` 定義（見 `CollectionEventId.cs`）。C# 會自動隱式轉換。

#### 2. U1 — 部分 VID 值 (如 CONTROLSTATE)

```
線上格式：<UI1>3</UI1>
```

```csharp
// secs4net 解析
byte state = vidList[1].FirstValue<byte>();  // V[1] = VID 2028 (CONTROLSTATE)
Console.WriteLine($"CONTROLSTATE = {state}");  // 輸出：CONTROLSTATE = 3

// 常見值對照
// 0 = Unknown, 1 = ATTEMPT_ONLINE, 2 = HOST_OFFLINE_RETRY
// 3 = HostOffline, 4 = OnlineLocal, 5 = OnlineRemote
```

#### 3. A (ASCII) — 設備時鐘、製程狀態名稱

```
線上格式：<ASC>2026041417355095</ASC>
```

```csharp
// secs4net 解析
string clock = vidList[0].GetString();  // V[0] = VID 2004 (Clock)
Console.WriteLine($"Clock = {clock}");   // 輸出：Clock = 2026041417355095

// 解析設備時鐘格式：yyyyMMddHHmmssff
// 2026-04-14 17:35:50.95 (UTC+8)
```

> **為什麼時鐘用 ASCII 而非數字？** 設備時鐘是複合欄位（日期+時間），用字串可保持精確的格式語義，避免時區和精度問題。SEMI 標準也建議時間相關 VID 使用 ASCII 格式。

#### 4. B (Binary) — S6F12 回覆 ACKC6

```
線上格式：<BIN>0</BIN>
```

```csharp
// secs4net 建立 S6F12 回覆
using var s6f12 = new SecsMessage(6, 12)
{
    Name     = "EventReportAck",
    SecsItem = Item.B(0x00),  // ACKC6 = 0 (accepted)
};
await wrapper.TryReplyAsync(s6f12, ct);
```

| ACKC6 值 | 語義 |
|----------|------|
| `0x00` | Accepted — 資料已接受 |
| `0x01` | Denied — 拒絕接受 |
| `0x02` | Busy — 忙碌中，稍後重試 |
| `0x03` | Invalid Data — 資料格式錯誤 |

#### 5. L (List) — 巢狀結構容器

```
線上格式：
<LST>
  <UI4>2</UI4>      ← RPTID
  <LST>             ← Values
    <ASC>...</ASC>
    <UI1>3</UI1>
  </LST>
</LST>
```

```csharp
// secs4net 解析報告列表
var reportList = root[2];                    // {L:a ...}
for (int i = 0; i < reportList.Count; i++)   // 迴圈 a
{
    var rpt = reportList[i];
    ulong rptid   = rpt[0].FirstValue<uint>();  // RPTID
    var   vidList = rpt[1];                     // {L:b ...}
    for (int j = 0; j < vidList.Count; j++)     // 迴圈 b
    {
        // 每個 vidList[j] 是一個 V 值，型別可能是 U1, U4, A, 或 L
        // 需要根據 VID 定義決定如何解析
    }
}
```

### 複雜 V 值範例：巢狀串列 (CEID=1004 MachineMessages)

某些 VID 的值**不是簡單純量**，而是巢狀串列：

```xml
<!-- V[0] 包含一個巢狀 L:3 -->
<LST>
  <LST>                          <!-- V[0] 第 1 個子元素：主動警報列表 -->
    <ASC>error,X30_G2_n</ASC>
    <ASC>error,199</ASC>
    <ASC>stop,2</ASC>
  </LST>
  <LST />                        <!-- V[0] 第 2 個子元素：已解除警報（空列表） -->
</LST>
```

```csharp
// secs4net 解析巢狀 V 值
var v0 = vidList[0];              // V[0] 是一個 L (List)
if (v0.Format == SecsFormat.List)
{
    var activeAlarms = v0[0];     // 第一個子串列：主動警報
    for (int k = 0; k < activeAlarms.Count; k++)
    {
        string alarmId = activeAlarms[k].GetString();
        // alarmId = "error,X30_G2_n", "error,199", "stop,2"
    }

    var clearedAlarms = v0[1];    // 第二個子串列：已解除警報
    // clearedAlarms.Count == 0（空列表 = 無已解除警報）
}
```

> **關鍵概念：** `[V]` 的型別標註為 `Any`，表示 SECS-II 允許**任何型別**出現在此位置。Host 端必須根據 VID 的語義定義（由設備廠商提供）來決定如何解析每個 V 值。這是 GEM 彈性的來源，也是實作複雜度的來源。

## secs4net 現代 C# 實作 (Modern secs4net Implementation)

本專案使用 **secs4net v2.4.4** 的 `Item` API 來處理 S6F11R，位於 `SecsApp/Handlers/S6F11Handler.cs`。

### Handler 完整架構

```csharp
public sealed class S6F11Handler : IMessageHandler
{
    public int Stream   => 6;     // 對應 S6F11
    public int Function => 11;    // Function 11

    // 訊號機制：當收到 CEID=2 (EquipmentOffline) 時通知外部等待者
    private readonly TaskCompletionSource _offlineEventReceived
        = new(TaskCreationOptions.RunContinuationsAsynchronously);
    public Task OfflineEventReceived => _offlineEventReceived.Task;

    public async Task HandleAsync(PrimaryMessageWrapper wrapper, CancellationToken ct)
    {
        // ... 解析與回覆邏輯（見下文）
    }
}
```

### 解析流程對照：舊版 vs. 新版

#### 舊版 (SecsPort.ListSplit — 字串解析)

```csharp
// 字串式解析：SecsPort.ListSplit() 回傳 string[]
string[] list0 = SecsPort.ListSplit(TSN_data);
if (list0.Length != 4 || list0[0] != "L:3") { ok = false; break; }

string[] list1 = SecsPort.ListSplit(list0[1]);  // DATAID
DATAID = list1[1];                                // 取得字串值

string[] list1 = SecsPort.ListSplit(list0[2]);  // CEID
CEID = list1[1];                                  // 取得字串值
```

#### 新版 (secs4net Item — 強型別物件)

```csharp
// 物件式解析：直接透過 Item 索引器和型別方法
var root = wrapper.PrimaryMessage.SecsItem;  // Item 物件

// 驗證：檢查是否為 List 且至少有 3 個子元素
if (root.Format != SecsFormat.List || root.Count < 3) { ... }

// 直接以強型別取出值
ulong dataid = root[0].FirstValue<uint>();  // DATAID — 自動從 U4 轉為 uint→ulong
ulong ceid   = root[1].FirstValue<uint>();  // CEID
```

**對照差異：**

| 面向 | 舊版 ListSplit | 新版 secs4net Item |
|------|---------------|-------------------|
| 型別 | `string[]`，所有值都是字串 | `Item` 物件，保留原始 SECS 型別 |
| 解析 | `list1[1]` 取出字串值 | `.FirstValue<uint>()` 取出強型別 |
| 驗證 | 檢查 `list0[0] == "L:3"` | 檢查 `root.Format == SecsFormat.List` |
| 巢狀 | 遞迴呼叫 `ListSplit()` | 直接用 `root[2][0]` 索引 |
| 錯誤 | 手動 `ok=false; break` | 例外或 Format 檢查 |

### 事件信號機制 (TaskCompletionSource Pattern)

`S6F11Handler` 展示了 GEM 通訊中常見的**跨訊息協調模式**：

```csharp
// Handler 內部：當 CEID=2 到達時觸發信號
if (ceid == CollectionEventId.EquipmentOffline)
{
    await ReplyAckAsync(wrapper, ct);
    _offlineEventReceived.TrySetResult();  // ← 通知等待者
    return;
}

// Program.cs 外部：等待設備進入 Offline
await s6f11Handler.OfflineEventReceived.WaitAsync(cts.Token);
```

**時序：**
```
Program.cs                    S6F11Handler              Equipment
    │                              │                         │
    ├── SendS1F15R ──────────────────────────────────────────→│ (請求 Offline)
    │                              │                         │
    │                              │←── S6F11R CEID=2 ──────│ (事件通知)
    │                              ├── S6F12 ACK ───────────→│
    │                              ├── TrySetResult()        │
    │←── OfflineEventReceived ─────┤                         │
    │  (繼續執行後續邏輯)           │                         │
```

## CEID (Collection Event ID) 詳細說明

**CEID** 是 **Collection Event Identifier**（集合事件識別碼）的縮寫，定義於 SEMI E30 (GEM) 規範。它代表設備上發生的一個「**可觀察事件**」。每當設備狀態改變或特定動作發生，就會觸發對應的 Collection Event，並透過 **S6F11** 將相關數據回報給主機 (Host)。

### 觸發流程

```
設備事件發生
     ↓
設備查找對應的 CEID
     ↓
收集該 CEID 綁定的 RPTID 數據
     ↓
發送 S6F11R → Host
```

### 典型的 CEID 範例

| CEID | 代表事件 |
|------|---------|
| `0`  | 控制狀態切換至 Remote (ControlStateRemote) |
| `1`  | 控制狀態切換至 Local (ControlStateLocal) |
| `2`  | 設備下線 (EquipmentOffline) |
| `9`  | 製程完成 (ProcessingCompleted) |
| `11` | 製程狀態變更 (ProcessingStateChange) |
| `100` | 單元處理完成 (UnitProcessed) — 廠商自定義 |
| `1004` | 機台訊息/警報 (MachineMessages) — 廠商自定義 |
| `1001` | 生產狀態 (ProductionState) — 廠商自定義 |

> 本專案的完整 36 個 CEID 定義見 `SecsApp/Protocol/CollectionEventId.cs`。

### CEID 的 GEM 設定生命週期

CEID 需要事先由 Host 透過其他訊息進行設定與啟用，才能收到 S6F11 事件報告。這個設定流程稱為 **GEM Event Setup Lifecycle**：

```
Phase 1: 建立通訊
  S1F13/F14  ── 建立連線 (Establish Communications)

Phase 2: 查詢能力
  S1F23/F24  ── 查詢設備支援哪些 CEID (Collection Event Namelist Request)

Phase 3: 定義報告模板
  S2F33/F34  ── 定義每個 RPTID 包含哪些 VID (Define Report)
  例如：RPTID=2 包含 VID 2004, 2028, 2033, 2034, 2035, 2036, 4030

Phase 4: 綁定事件與報告
  S2F35/F36  ── 將 CEID 與 RPTID 綁定 (Link Report to Collection Event)
  例如：CEID=2 (EquipmentOffline) → RPTID=2

Phase 5: 啟用事件
  S2F37/F38  ── 啟用/停用特定 CEID (Enable/Disable Collection Event)
  例如：啟用 CEID=0, 1, 2, 9, 11, 100, ...

Phase 6: 等待事件（正常運行）
  當事件發生時，設備自動發送 S6F11R
```

> **重要：** S6F11R 的資料內容完全取決於 Phase 3–5 的設定。同一個 CEID，不同的 RPTID 綁定會產生不同的 V 值列表。Host 必須維護自己的 CEID→RPTID→VID 映射表才能正確解析。

### CEID 在 S6F11 訊息中的解析（C#）

```csharp
// list0[2] 即為 CEID 元素，位於 L:3 的第二個子元素
list1 = SecsPort.ListSplit(list0[2]);
if (list1.Length < 1) { ok = false; break; }
if (list1.Length == 2) { CEID = list1[1]; }  // 取出實際值，例如 "2" (EquipmentOffline)
else { CEID = ""; }

// 取得 CEID 後，可依此判斷業務邏輯
// switch (CEID) {
//   case "1": HandleWaferLoad(); break;
//   case "2": HandleWaferUnload(); break;
// }
```

### 正確的 Host 端防禦性處理（C#）

當 Host 收到 S6F11R 時，應先檢查當前是否正在進行 OFF-LINE 流程：

```csharp
void recv_S6F11R(object sender, int stream, int function, bool send_reply, int transID, string TSN_data, string header) {
    SecsHost sp = (SecsHost)sender;
    bool ok = true;
    while (ok) {
        // [防禦] 若 Host 正在等待 Equipment 進入 OFF-LINE，忽略此事件報告
        if (IsExpectingOffline()) {
            if (send_reply) {
                // 回傳非零表示不接受（可選：依廠商規範決定是否應答）
                sp.SendReply(6, 12, transID, "B:1 0x01");
            }
            return;
        }

        // 正常解析 S6F11R ...
        string DATAID, CEID, RPTID, V;
        string[] list0 = SecsPort.ListSplit(TSN_data);
        if (list0.Length != 4 || list0[0] != "L:3") { ok = false; break; }
        // ... (略，同標準解析流程)

        if (send_reply) {
            sp.SendReply(6, 12, transID, "B:1 0x00");
        }
        return;
    } // end while(ok)
} // end recv_S6F11R
```

### 推薦的 OFF-LINE 流程

為避免此類 race condition，建議 Host 在送出 S1F15R 前，先透過 S2F37 停用所有 Collection Event：

```
Host → S2F37 (CEID list, disable all)  →  Equipment
Host ← S2F38 (DRACK=0)                 ←  Equipment
Host → S1F15R                          →  Equipment
Host ← S1F16 (OFLACK=0)               ←  Equipment
（此時不會再有 S6F11 出現）
```

## 後續分析：廠商說法 vs. 獨立判讀（含修正背景說明）

### 重要背景：此 log 為修正後資料

> **廠商確認：** RPTID=2 原本定義時確實**缺少 VID 2004 (Clock)**。廠商在發現問題後，已將 VID 2004 補加至 RPTID=2 的定義中。上述 log 是**修正後**擷取的訊息，因此 V[0] 已可見時間值。

> 這也是初次判讀時出現混亂的原因：判讀者看到的是修正後的 log（有時間值），但廠商描述的是修正前的問題（缺少時間 VID）——兩者描述的是**不同時間點的狀態**。

### 廠商回報的說法（事後確認正確）

> **「Link Report ID 缺少時間的 VID」**

> 廠商在原始 log 發生時，RPTID=2 的定義中**未包含 VID 2004 (Clock)**，導致 S6F11R 回傳的 V 值列表缺少時間欄位。此問題已被廠商識別並修正（補加 VID 2004）。

### 修正後仍存在的 Issue 說明

VID 補加修正了時間欄位缺失的問題。其餘觀察到的 V 值均為正常：

#### S6F11R (CEID=2) 出現在 S1F16 之後：**此順序正確**

> **注意：** 先前曾誤判此順序為「顛倒」，經重新審視 GEM 狀態機語義後確認此順序是**正確的**，特此說明。

GEM 狀態機的邏輯是：

1. Equipment 收到 S1F15R，執行內部狀態轉換（進入 HostOffline）
2. 狀態轉換完成後，送出 S1F16 (OFLACK=0)——語義為「**我已**進入 OFF-LINE」
3. 狀態轉換完成同時觸發 CEID=2 (EquipmentOffline)，送出 S6F11R——語義為「**通知**此狀態變更事件」

```
09:36:22.553  Host      → S1F15R              （請求 OFF-LINE）
09:36:22.572  Equipment → S1F16 OFLACK=0      （已完成狀態轉換）✅
09:36:22.582  Equipment → S6F11R CEID=2       （通知 EquipmentOffline 事件）✅
09:36:22.595  Host      → S6F12 BIN=0         （ACK）✅
```

#### CommState = 260 (communicating) 的說明：**正常，非矛盾**

> **注意：** 先前曾誤判 `CommState=260` 與 `CONTROLSTATE=3 (HostOffline)` 為「語義矛盾」，此判斷有誤，特此更正。

| VID | 名稱 | 值 | 說明 |
|-----|------|----|------|
| 2028 | CONTROLSTATE | `3` = HostOffline | GEM **控制狀態層**：已切換為 Host 要求的下線狀態 ✅ |
| 2036 | CommState | `260` = communicating | GEM **通訊狀態層**：HSMS 連線仍維持 ✅ |

`CONTROLSTATE` 與 `CommState` 是兩個**不同層次**的狀態機：
- **CONTROLSTATE** 描述 GEM 控制層（Online/Offline/Local/Remote）
- **CommState** 描述 HSMS 通訊層（連線是否建立）

Host 請求 OFF-LINE 只會切換控制層狀態，**不會斷開 HSMS 連線**。設備在送出 S6F11R 時通訊層仍為 `communicating` 是物理上必然的（有連線才能送出訊息），兩個值同時存在完全正常。

```xml
        </LST>
    </SECSData>
</SECSMessage>
```

### S6F11R 資料結構解析

| 欄位 | 值 | 說明 |
|------|----|------|
| DATAID | 8 | 訊息關聯 ID |
| CEID | 11 | `ProcessingStateChange` — 廠商自定義製程狀態切換事件 |
| RPTID | 11 | `ProcessingStateChange` 報告模板（4 個 VID） |
| V[0] | `"2026033116204250"` (ASC) | VID 2004 (Clock)：設備時鐘，UTC+8 = 16:20:42.50 |
| V[1] | 4 (UI1) | VID 2030 (PREVIOUSPROCESSSTATE)：前一個製程狀態 = 4 |
| V[2] | 1 (UI1) | VID 2031 (PROCESSSTATE)：當前製程狀態（數字）= 1 |
| V[3] | `"PRODUCTION"` (ASC) | VID 2032 (ProcessState)：當前製程狀態（名稱）|

### RPTID=11 VID 綁定（規格定義）

| 位置 | VID | VID 名稱 | 型別 | 說明 |
|------|-----|---------|------|------|
| V[0] | 2004 | Clock | A | 設備時鐘 |
| V[1] | 2030 | PREVIOUSPROCESSSTATE | U1 | 上一個製程狀態（數字）|
| V[2] | 2031 | PROCESSSTATE | U1 | 當前製程狀態（數字）= 1 = 生産中 |
| V[3] | 2032 | ProcessState | A | 當前製程狀態（名稱）= `"PRODUCTION"` |

### Issue 分析

| # | 嚴重度 | 問題 | 說明 |
|---|--------|------|------|
| 1 | 🔴 High | CEID=11 而非規格要求 CEID=1001 | 設備使用廠商自定義 `ProcessingStateChange` 代替 GEM 標準 `ProductionState`，CEID 與規格不符 |
| 2 | 🟢 Info | Report 結構完整，VID 綁定正確 | RPTID=11 包含 VID 2004/2030/2031/2032，四個欄位均存在且型別正確 |

> **注意：** VID 2030 (PREVIOUSPROCESSSTATE) 與 VID 2028 (CONTROLSTATE) 雖然值相同（均為 4），但語義不同：前者表示「切換前的製程狀態為 4（WAIT）」，後者表示「GEM 控制狀態」。RPTID=11 不包含 VID 2028，切勿混淆。

## 三個案例的型別使用對比

下表整理三個實戰案例中出現的所有 SECS 資料型別，展示同一個 S6F11R 格式如何承載完全不同的資料內容：

| 案例位置 | SECS 型別 | 欄位 | 值 | 語義 |
|---------|-----------|------|----|------|
| 案例一 V[0] | **ASC** | Clock | `"2026041417355095"` | 設備時鐘 (UTC+8) |
| 案例一 V[1] | **UI1** | CONTROLSTATE | `3` | HostOffline |
| 案例一 V[5] | **UI4** | CommState | `260` | Communicating |
| 案例二 V[0] | **巢狀 L** | MachineMessages | `{ {errors...}, {} }` | 主動/已解除警報列表 |
| 案例三 V[2] | **UI1** | PROCESSSTATE | `1` | 製程狀態數字 |
| 案例三 V[3] | **ASC** | ProcessState | `"PRODUCTION"` | 製程狀態名稱 |

> **核心要點：** S6F11R 的 `[V]` 標註為 `Any` type。同一個訊息格式中，不同的 CEID/RPTID 組合會產生**完全不同型別和語義**的 V 值。Host 端必須維護 VID 定義表，才能正確解析每個欄位。這就是 GEM Collection Event Framework 的設計精髓——**一個格式，無限場景**。
