# S1F15R - Request OFF-LINE

**Direction:** Sent by Host Only

---

| **S1F15R** | Request OFF-LINE | Sent by Host Only |

Format:

_header only_

> **說明：** 這個訊息沒有 data body（資料本體），只有 SECS-II 訊息標頭（header）。在 SECS-II 協定中，每個訊息由固定 10 bytes 的 header（包含 stream、function、W-bit、transaction ID 等）加上可選的 data body 組成。`header only` 表示 S1F15R 是一個**空訊息**，Host 只發出標頭，不附帶任何資料。Equipment 收到後不需要解析 TSN data（程式碼中反映為 `// no data expected`），直接準備回覆 S1F16（OFF-LINE Acknowledge）。

```
    // S1F15R C# Receive message - add next line to setup
    //SecsPort sp=your_SecsPort;
    sp.MessageTypeAdd( 1, 15, new SecsMsgReceiveDelegate(recv_S1F15R));

// receive method
void recv_S1F15R(object sender, int stream, int function, bool send_reply, int transID, string TSN_data, string header) {
    SecsPort sp = (SecsPort)sender;
    bool ok=true;
    while (ok) {   // break out of loop on error
        // no data expected
        if (send_reply) {
            string reply; // TBD create reply
            sp.SendReply(1, 16, transID, reply);
            }
        return;
        } // end while(ok)
    // bad data
    sp.SendS9(7, header);
    } // end recv_S1F15R
```

**Codebase 實現說明：**
- **`sp.MessageTypeAdd(1, 15, new SecsMsgReceiveDelegate(recv_S1F15R))`** — 註冊訊息處理器，當 Equipment 收到 S(1)F(15) 時呼叫 `recv_S1F15R` 方法
- **`while (ok)` 迴圈** — secs4net 的錯誤處理慣例，正常結束用 `return`，出錯用 `break` 跳出
- **`// no data expected`** — 因為 Format 是 header only，不需要解析 TSN_data
- **`if (send_reply)`** — 檢查是否需要回覆，是則呼叫 `sp.SendReply(1, 16, transID, reply)` 回覆 S1F16（OFF-LINE Acknowledge）
- **`sp.SendS9(7, header)`** — 若資料驗證失敗，回覆 S9F7（illegal data）錯誤訊息給 Host

## 實戰案例：S1F15R 交握流程與 S6F11R 時序觀察

### 場景描述

以下 XML 訊息序列記錄了一次真實的 S1F15R 交握流程：

```xml
<!-- 1. Host 送出 OFF-LINE 請求 -->
<SECSMessage s="1" f="15" direction="H to E" replyBit="true"
             txid="2000265" time="2026-04-14T09:36:22.553Z" deviceID="0">
    <SECSData />
</SECSMessage>

<!-- 2. Equipment 完成狀態轉換後回傳 OFLACK=0 -->
<SECSMessage s="1" f="16" direction="E to H" replyBit="false"
             txid="2000265" time="2026-04-14T09:36:22.572Z" deviceID="0">
    <SECSData><BIN>0</BIN></SECSData>
</SECSMessage>

<!-- 3. Equipment 觸發 EquipmentOffline Collection Event -->
<SECSMessage s="6" f="11" direction="E to H" replyBit="true"
             txid="9" time="2026-04-14T09:36:22.582Z" deviceID="0">
    ...
</SECSMessage>
```

### 時序分析

| 時間戳記 | 訊息 | 說明 |
|---------|------|------|
| `09:36:22.553` | Host → S1F15R | Equipment 收到 OFF-LINE 請求 |
| `09:36:22.572` | Equipment → S1F16 (OFLACK=0) | 狀態轉換完成，確認進入 HostOffline |
| `09:36:22.582` | Equipment → S6F11R (CEID=2) | ✅ EquipmentOffline 事件通知，**此順序正確** |

### 關於 S6F11R 在 S1F16 之後的說明

S6F11R (CEID=2 EquipmentOffline) 出現在 S1F16 之後 10ms，**此順序符合 GEM 規範**：

- S1F16 (OFLACK=0) 語義為「**我已**完成狀態轉換，進入 OFF-LINE」
- Collection Event CEID=2 (`EquipmentOffline`) 在狀態轉換**完成後**才被觸發
- 因此 S6F11R 必然在 S1F16 之後送出，是正確行為

### 此案例中真正的 Issue

實際問題為 **RPTID=2 原始定義缺少 VID 2004 (Clock)**，導致 S6F11R 回傳的 V 值列表中缺少時間欄位。問題已被廠商識別並修正（補加 VID 2004）。詳見 `S6F11R.md` 中的完整分析。
