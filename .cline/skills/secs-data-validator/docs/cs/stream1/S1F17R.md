# S1F17R - Request ON-LINE

**Direction:** Sent by Host Only

---

| **S1F17R** | Request ON-LINE | Sent by Host Only |
| --- | --- | --- |

## 用意與目的

`S1F17R` 是 SECS-II Stream 1 的控制訊息，用於 **Host（主機/上位機）主動要求 Equipment（設備）進入 ON-LINE 狀態**。

### 使用情境

| 情境 | 說明 |
|------|------|
| 設備重新上線 | 設備曾因維護或異常進入 OFF-LINE，Host 發出 S1F17R 要求重返 ON-LINE |
| 自動化啟動流程 | Host 在建立 HSMS 連線後，依 GEM 狀態機進行 ON-LINE 切換 |
| 遠端控制 | Operator 透過 Host 系統介面下令設備上線 |

### 訊息方向

```
Host ──[S1F17R]──▶ Equipment
Host ◀──[S1F18]── Equipment
```

- S1F17R：Host 送出，**R = Reply bit 為 true**，表示 Host 期待回覆
- S1F18：Equipment 回傳，攜帶 `ONLACK (B:1)` 表示是否接受上線請求

### 與 S1F15R 的對比

| 訊息 | 目的 | 回覆 | 動作 |
|------|------|------|------|
| S1F15R | 要求設備進入 **OFF-LINE** | S1F16 (OFLACK) | Equipment 停止生產接受管理 |
| S1F17R | 要求設備進入 **ON-LINE** | S1F18 (ONLACK) | Equipment 恢復正常生產狀態 |

## 資料格式

Format:

_header only_

> **說明：** 這個訊息沒有 data body（資料本體），只有 SECS-II 訊息標頭（header）。
>
> 在 SECS-II 協定中，每個訊息由固定 10 bytes 的 header（包含 stream、function、W-bit、transaction ID 等欄位）加上**可選**的 data body 組成。
>
> - `header only` = 只有標頭，不附帶任何 TSN data
> - 接收端不需要解析任何資料（程式碼中反映為 `// no data expected`）
> - 收到請求後直接準備回覆 S1F18（ON-LINE Acknowledge）

### 為何 S1F17R 不需要資料？

S1F17R 是一個**純命令訊息（pure command message）**，其語義為「請進入 ON-LINE 狀態」，本身不需要附帶參數。命令的接受或拒絕由 Equipment 在 S1F18 的 `ONLACK` 欄位中回應。這類 header-only 設計在 SECS-II 中很常見，例如 S1F1R（Are You Online?）和 S1F15R（Request OFF-LINE）也是相同結構。

---

## 回覆訊息：S1F18 (ON-LINE Acknowledge)

Equipment 收到 S1F17R 後，**必須**回傳 S1F18，其格式為：

```
S1F18 = ONLACK (B:1)
```

| ONLACK 值 | 含義 |
|-----------|------|
| `0x00` | OK — 設備已成功進入 ON-LINE 狀態 |
| 非零值 | 拒絕或失敗 — 設備無法立即上線（參考廠商文件） |

## 舊式 API 範例（SecsPort）

以下為傳統 `SecsPort` 風格的 C# 接收處理，對應文件中常見的 secs-cslib 格式：

```csharp
// S1F17R C# Receive message - add next line to setup
//SecsPort sp=your_SecsPort;
sp.MessageTypeAdd(1, 17, new SecsMsgReceiveDelegate(recv_S1F17R));

// receive method
void recv_S1F17R(object sender, int stream, int function, bool send_reply, int transID, string TSN_data, string header) {
    SecsPort sp = (SecsPort)sender;
    bool ok = true;
    while (ok) {   // break out of loop on error
        // no data expected — S1F17R is header only
        if (send_reply) {
            // S1F18 reply: ONLACK (B:1) = 0x00 means OK
            string reply = "B:1 0x00";
            sp.SendReply(1, 18, transID, reply);
        }
        return;
    } // end while(ok)
    // bad data
    sp.SendS9(7, header);
} // end recv_S1F17R
```

> **舊式 API 說明：**
>
> **註冊：** `sp.MessageTypeAdd(1, 17, ...)` 告訴 SECS port 物件 `sp`，當收到 S1F17R 訊息時呼叫 `recv_S1F17R` handler。
>
> **Handler 參數：**
> - `sender` — 觸發此事件的 SECS port 物件（cast 回 `SecsPort`）
> - `stream` / `function` — 訊息的 S/F 編號（此處固定 1, 17）
> - `send_reply` — 是否需要回覆（Host 發出的是 Primary，Equipment 需回 S1F18）
> - `transID` — 交易編號，回覆時必須帶同一個 ID
> - `TSN_data` — TSN 格式的資料（此訊息為 header only，不會有資料）
> - `header` — 原始 SECS-II header 字串
>
> **Handler 邏輯：**
> 1. `while (ok)` — 以 break-on-error 模式處理，正常流程在末尾 `return`
> 2. `// no data expected` — 不需要解析 TSN_data
> 3. `sp.SendReply(1, 18, transID, reply)` — 回覆 S1F18（ON-LINE Acknowledge），reply = `"B:1 0x00"` 表示接受
> 4. 若資料有誤，跳出 loop 後執行 `sp.SendS9(7, header)` — 發送 S9F7（illegal data）錯誤訊息

## secs4net 現代 C# 實作（IMessageHandler）

以下為本 codebase 所使用的 secs4net v2.4.4 風格，與 `S1F1Handler.cs` 的架構一致：

```csharp
// SecsApp/Handlers/S1F17Handler.cs
using Secs4Net;
using static Secs4Net.Item;

namespace SecsApp.Handlers;

public sealed class S1F17Handler : IMessageHandler
{
    public int Stream   => 1;
    public int Function => 17;

    public async Task HandleAsync(PrimaryMessageWrapper wrapper, CancellationToken ct)
    {
        // S1F17R is header-only — no data to parse.
        // Reply S1F18 with ONLACK = B:1 (0x00 = OK)
        using var reply = new SecsMessage(1, 18)
        {
            Name     = "OnlineAcknowledge",
            SecsItem = B(0x00), // ONLACK: 0 = OK, transition to ON-LINE accepted
        };

        bool sent = await wrapper.TryReplyAsync(reply, ct);
        Console.WriteLine($"[TX] S1F18 (ONLACK=0x00, ON-LINE accepted) → {(sent ? "OK" : "Already replied")}");
    }
}
```

### 資料型態說明：`Item.B(byte)` — Binary

| 概念 | 說明 |
|------|------|
| **SECS-II 型態** | `B` = Binary，每個元素為 1 byte（`0x00`–`0xFF`） |
| **secs4net 工廠方法** | `Item.B(params byte[] values)` |
| **單一 byte** | `B(0x00)` → `B:1 [0x00]` |
| **多個 bytes** | `B(0x00, 0x01, 0xFF)` → `B:3 [0x00, 0x01, 0xFF]` |
| **ONLACK 慣例** | 值 `0` = 成功，非零 = 拒絕/錯誤 |

```csharp
// Binary 型態的各種用法範例
Item onlack_ok    = Item.B(0x00);        // ONLACK = 0, 接受上線
Item onlack_error = Item.B(0x01);        // ONLACK = 1, 拒絕（設備忙碌或不允許）
Item multi_bytes  = Item.B(0x00, 0x01);  // B:2 — 多 byte 範例（S1F18 不用，僅示意）
```

> **為何 S1F18 使用 `B:1` 而非 `U1:1`（1-byte unsigned integer）？**
>
> SECS-II 標準明確定義 `ONLACK` 為 Binary 型態（`B`），而非數值型態（`U1`/`I1`）。
> 在協定設計上，Binary 用於**狀態碼與旗標（flag）**，數值型態用於**可計算的量**。
> 接收方必須嚴格匹配型態，否則應視為格式錯誤（S9F7）。

### 與 `S1F1Handler` 的結構對比

```
S1F1Handler（header only → 回 L:0）        S1F17Handler（header only → 回 B:1）
─────────────────────────────────────     ─────────────────────────────────────
Stream = 1, Function = 1                  Stream = 1, Function = 17
no data to parse                          no data to parse
reply = SecsMessage(1, 2) { L() }         reply = SecsMessage(1, 18) { B(0x00) }
```

兩者結構完全平行，差異只在回覆的 S/F 編號與 data item 型態。

## 完整交握流程示意

```
Host                                Equipment
 │                                       │
 │── S1F17R (header only, W-bit=1) ─────▶│
 │                                       │ 檢查是否可以進入 ON-LINE
 │◀── S1F18 (ONLACK=B:1 [0x00]) ─────── │ ONLACK=0 表示接受
 │                                       │
 │ Host 確認 ONLACK == 0                 │ Equipment 狀態 → ON-LINE
 │                                       │
```

### Host 端解析 S1F18 回覆的範例

以下展示 Host 如何在送出 S1F17R 後解析 Equipment 回傳的 S1F18：

```csharp
// OutboundCommands 模式：Host 送出 S1F17R 並等待 S1F18
var request = new SecsMessage(1, 17) { Name = "RequestOnline" };
// S1F17R 為 header only，SecsItem 保持 null

SecsMessage reply = await secsGem.SendAsync(request, ct);

// 解析 S1F18 的 ONLACK (B:1)
if (reply.SecsItem is null || reply.SecsItem.Format != SecsFormat.Binary)
    throw new InvalidOperationException("S1F18: expected ONLACK (B:1)");

byte onlack = reply.SecsItem.FirstValue<byte>();
if (onlack != 0)
    throw new InvalidOperationException($"S1F18: ONLACK={onlack}, equipment refused online request");

Console.WriteLine("[RX] S1F18: ONLACK=0, equipment transitioned to ON-LINE");
```

## 在 Program.cs 中註冊 Handler

```csharp
// Program.cs — 加入以下行至 handler 註冊區段
factory.Register(new S1F17Handler());
```

這告訴 `MessageHandlerFactory`，當 `MessageDispatcher` 收到 Stream=1, Function=17 的訊息時，
呼叫 `S1F17Handler.HandleAsync()`。
