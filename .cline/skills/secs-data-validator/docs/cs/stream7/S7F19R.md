# S7F19R - Current Process Program Dir Request

**Direction:** Host Only

---

| **S7F19R** | Current Process Program Dir Request | Host Only |

### 用意與目的

S7F19R（**Current Process Program Dir Request**）是 SECS-II Stream 7（製程程式管理）中的**配方目錄查詢訊息**。Host 用此訊息向設備詢問：「你現在有哪些製程程式？」設備以 S7F20 回覆，包含目前所有已儲存在設備上的 PPID 清單。

#### 在 GEM 工作流程中的定位

```
Host                                Equipment
  │                                     │
  │─── S7F19R (header only) ───────────▶│  ← 本訊息：查詢配方目錄
  │◀─── S7F20 (L:n { PPID... }) ────────│  設備回傳所有 PPID 清單
  │                                     │
  │  Host 得知設備上現有的配方清單         │
  │  可做比對、上傳、刪除或稽核等後續動作   │
```

S7F19R 屬於 **SEMI GEM（E30）規範 Required GEM Capability #7：Process Program Management** 的必要組成部分。典型使用情境：

- **配方稽核：** Host 定期查詢設備配方清單，確認是否有未授權的配方存在，或期望的配方是否遺失。
- **配方上傳前確認：** Host 在 S7F23R 上傳新配方前，先確認設備是否已有同名 PPID，避免覆寫衝突。
- **配方刪除管理：** Host 使用 S7F17R 刪除配方前，先用 S7F19R 確認目標 PPID 確實存在。
- **設備換線/換型：** 換產時 Host 查詢設備上現有的配方，再決定需要上傳哪些、刪除哪些。
- **MES 同步：** MES 系統啟動或重連時，拉取設備配方清單，與本地資料庫對齊。

#### 訊息名稱解析

| 部分 | 含義 |
|------|------|
| `S7` | Stream 7 — 製程程式管理 |
| `F19` | Function 19 — 第 19 號功能（Request） |
| `R` | Reply requested（需要回覆）— Host 期待收到 S7F20 作為清單回應 |

> **為何是 header only？** S7F19R 不需要任何輸入參數 — Host 只是問「你有什麼配方？」，不需要指定 PPID 或其他條件。所有資訊（Stream=7, Function=19, W-bit=1）都編碼在 10 個位元組的 HSMS 訊息標頭裡。secs4net 中此訊息的 `SecsItem` 為 `null`，接收端絕不應嘗試讀取其內容。

#### 與相關訊息的關係

| 訊息 | 方向 | 用途 |
|------|------|------|
| **S7F19R** | Host → Equipment | 請求目前配方目錄（本文件） |
| S7F20 | Equipment → Host | 回傳所有 PPID 的清單（`L:n { PPID }`） |
| S7F17R | Host → Equipment | 刪除指定 PPID 的配方 |
| S7F23R | Host → Equipment | 上傳格式化配方至設備（以 PPID 為鍵） |
| S7F25R | Host ↔ Equipment | 請求取回格式化配方內容（以 PPID 為鍵） |
| S7F3R | Host ↔ Equipment | 傳送非格式化製程程式（PPID + PPBODY） |
| S7F5R | Host ↔ Equipment | 請求取回非格式化製程程式（以 PPID 為鍵） |

### Format (TSN)

```
header only
```

**Format Explanation:**

這是一個**純標頭訊息（header-only message）**，不帶任何 data item：

- **無資料項目** — S7F19R 的訊息本體（body）為空，secs4net 中 `SecsItem` 屬性為 `null`。
- 所有資訊都編碼在 HSMS 訊息標頭中（Stream=7, Function=19, W-bit=1），不需要解析任何資料。

> **注意：** 在 secs4net 中，header-only 訊息與 `L:0`（空 list）是不同的。`SecsMessage(7, 19)` 的 body 為零位元組，而 `SecsMessage(7, 19, Item.L())` 會在 wire 上多編碼 2 個位元組的 `L:0`。接收端的 `handler` 不應嘗試存取 `wrapper.PrimaryMessage.SecsItem`（它為 `null`）。

### 資料型別說明與範例

S7F19R 本身無資料項目，回覆 S7F20 的資料項目如下：

#### S7F20 回覆格式

```
{L:n
  [PPID]
}
```

| 欄位 | SECS 型別 | 長度限制 | 說明 |
|------|-----------|----------|------|
| `L:n` | List | n = 0 ～ 上限由設備定義 | 配方清單，n 為設備上製程程式的數量 |
| `PPID` | `A` (ASCII) | 最大 80 字元 | 製程程式 ID，每個元素一個 PPID |

#### 回覆範例

| 場景 | S7F20 內容 | 說明 |
|------|-----------|------|
| 無配方 | `L:0` | 設備上沒有任何製程程式 |
| 有 1 個配方 | `L:1 { A "RECIPE_GOLD" }` | 設備上只有一個配方 |
| 有 3 個配方 | `L:3 { A "ETCH_5NM" } { A "CVD_OXIDE" } { A "DEFAULT" }` | 設備上有 3 個配方 |

**secs4net 建立 S7F19R（Host 端發送）：**

```csharp
// 查詢設備上現有的所有配方名稱
var msg = new SecsMessage(7, 19, replyExpected: true);
// 注意：不傳入任何 Item（header-only）
```

**SML 表示法：**

```
S7F19R
.
```

**secs4net 讀取收到的 S7F19R（設備端 handler）：**

```csharp
public async Task HandleAsync(PrimaryMessageWrapper wrapper)
{
    // S7F19R 是 header-only — 不需要也不應讀取 SecsItem
    // 直接建立 S7F20 回覆，包含所有已知的 PPID 清單
    var ppids = _recipeStore.GetAllPpids();
    await wrapper.TryReplyAsync(BuildS7F20(ppids));
}

private static SecsMessage BuildS7F20(IEnumerable<string> ppids) =>
    new SecsMessage(7, 20,
        Item.L(ppids.Select(p => Item.A(p)).ToArray()));
```

#### 完整互動範例：Host 查詢並解析配方清單

```csharp
// Host 端：發送 S7F19R 並等待 S7F20
public async Task<IReadOnlyList<string>> RequestProcessProgramDirectoryAsync(
    ISecsGem secsGem, CancellationToken ct = default)
{
    // header-only — 不帶任何 body
    var request = new SecsMessage(7, 19, replyExpected: true);

    SecsMessage reply = await secsGem.SendAsync(request, ct);

    // S7F20 結構：L:n { PPID... }
    var body = reply.SecsItem;
    if (body is null)
        return Array.Empty<string>();   // L:0 等同無配方

    var ppids = new List<string>(body.Count);
    for (int i = 0; i < body.Count; i++)
        ppids.Add(body[i].GetValue<string>());

    return ppids;
}
```

#### 正常流程（設備有配方）

```
Host                                Equipment
  │                                     │
  │─── S7F19R ──────────────────────────▶│
  │    (header only, no body)           │  設備查詢本地配方清單
  │                                     │
  │◀────────────── S7F20 ───────────────│
  │    L:3 {                            │
  │      A "ETCH_RECIPE_5NM"            │

  │    }                                │
```

#### 設備無配方（空清單）

```
Host                                Equipment
  │                                     │
  │─── S7F19R ─────────────────────────▶│
  │                                     │
  │◀── S7F20 (L:0) ─────────────────────│  ← 空 list，表示無任何配方
```

### C# secs4net 實作

#### 設備端：接收 S7F19R 並回覆 S7F20

```csharp
// Handlers/S7F19Handler.cs
using Secs4Net;

public sealed class S7F19Handler : IMessageHandler
{
    public int Stream   => 7;
    public int Function => 19;

    private readonly IRecipeStore _recipeStore;

    public S7F19Handler(IRecipeStore recipeStore)
        => _recipeStore = recipeStore;

    public async Task HandleAsync(PrimaryMessageWrapper wrapper)
    {
        // S7F19R 是 header-only — 無需也不應讀取 SecsItem
        var ppids = _recipeStore.GetAllPpids();
        var reply = BuildS7F20(ppids);

        await wrapper.TryReplyAsync(reply);
    }

    private static SecsMessage BuildS7F20(IEnumerable<string> ppids) =>
        new SecsMessage(7, 20,
            Item.L(ppids.Select(p => Item.A(p)).ToArray()));
}
```

#### Host 端：發送 S7F19R 並讀取 S7F20 清單

```csharp
// OutboundCommands.cs（Host 端）
public async Task<IReadOnlyList<string>> SendS7F19AndWaitS7F20Async(
    CancellationToken ct = default)
{
    var request = new SecsMessage(7, 19, replyExpected: true);
    SecsMessage reply = await _secsGem.SendAsync(request, ct);

    var body = reply.SecsItem;
    if (body is null) return Array.Empty<string>();

    var ppids = new List<string>(body.Count);
    for (int i = 0; i < body.Count; i++)
        ppids.Add(body[i].GetValue<string>());

    Console.WriteLine($"[S7F19R] Equipment has {ppids.Count} process programs:");
    foreach (var ppid in ppids)
        Console.WriteLine($"  - {ppid}");

    return ppids;
}
```

### PPSelect 後驗證：S2F41 → S2F42 → S7F19R 完整流程

在 GEM 換線（Job Change / Recipe Change）場景中，Host 先用 S2F41R 下達 `PP_SELECT` 命令，再接著用 S7F19R 確認目標配方確實存在於設備上。

#### 為什麼 PPSelect 後要查 S7F19R？

| 原因 | 說明 |
|------|------|
| **HCACK=0 ≠ 配方存在** | S2F42 的 `HCACK=0` 只表示設備接受了命令，不保證指定的 PPID 存在 |
| **防禦性驗證** | 確認設備配方庫與 MES 預期一致，及早發現「配方遺失」問題 |
| **換線前置確認** | 在物料進站（S7F7R）前確認配方清單，避免生產中斷 |
| **稽核合規** | 部分 fab 規範要求在換型後必須記錄設備上的配方清單 |

#### 通訊時序圖

```
Host                                Equipment
  │                                     │
  │─── S2F41R ─────────────────────────▶│  PP_SELECT PPID="ETCH_5NM"
  │    L:2 {                            │
  │      A "PP_SELECT"                  │
  │      L:1 {                          │
  │        L:2 { A "PPID"               │
  │               A "ETCH_5NM" }        │
  │      }                              │
  │    }                                │
  │                                     │
  │◀─── S2F42 ──────────────────────────│  HCACK=0（命令被接受）
  │    L:2 { B 0x00, L:0 }             │
  │                                     │
  │  [設備內部執行 PP_SELECT]            │  設備切換至 ETCH_5NM
  │                                     │
  │─── S7F19R (header only) ───────────▶│  ← 確認 ETCH_5NM 確在清單中
  │                                     │
  │◀─── S7F20 ──────────────────────────│
  │    L:3 {                            │
  │      A "ETCH_5NM"    ← ✓ 找到目標  │
  │      A "CVD_OXIDE"                  │
  │      A "DEFAULT"                    │
  │    }                                │
  │                                     │
  │  Host 驗證 PPID 存在後，可繼續換線流程│
```

#### C# secs4net 實作：PPSelect + S7F19R 驗證

```csharp
// OutboundCommands.cs（Host 端）
public static async Task<bool> SendPPSelectAndVerifyAsync(
    ISecsGem secsGem,
    string targetPpid,
    CancellationToken ct = default)
{
    // Step 1: S2F41R — 下達 PP_SELECT 命令
    await SendS2F41AndWaitS2F42Async(
        secsGem,
        rcmd: "PP_SELECT",
        parameters: [("PPID", targetPpid)],
        ct: ct);

    // Step 2: S7F19R — 查詢設備配方清單，確認目標 PPID 存在
    var ppids = await SendS7F19AndWaitS7F20Async(secsGem, ct);

    bool found = ppids.Contains(targetPpid, StringComparer.Ordinal);
    Console.WriteLine(found
        ? $"[PPSelect] Verified: PPID \"{targetPpid}\" is present on equipment."
        : $"[PPSelect] WARNING: PPID \"{targetPpid}\" NOT found after PP_SELECT!");

    return found;
}
```

> **注意：** GEM 標準亦定義了 CEID `PPSelected` (CEID=8)。設備完成 PP_SELECT 後，可能非同步發出 `S6F11` 事件通知。若要避免 race condition，建議等待 S6F11/CEID=8 之後再發 S7F19R，或在 `S6F11Handler` 中觸發後續的查詢動作。

### S7F25R Round-trip 驗證：上傳後讀回確認配方完整性

在 Host 完成 S7F23R 配方上傳後，使用 S7F25R 讀回設備上儲存的配方內容，逐欄位比對以確認傳輸正確。這是一種「write → read-back → compare」模式，確保配方沒有因傳輸錯誤或設備端轉換而失真。

#### 為什麼需要 Round-trip 驗證？

| 原因 | 說明 |
|------|------|
| **傳輸可靠性** | HSMS 雖有 checksum，但設備端的儲存或轉換邏輯可能修改配方內容 |
| **格式正規化** | 設備可能在儲存時對浮點數或字串做正規化，讀回後與上傳值有微小差異 |
| **MDLN / SOFTREV 相容性** | S7F26 回傳的 MDLN、SOFTREV 可確認配方是否適用於此設備型號 |
| **稽核留證** | 部分 fab 規範要求在配方生效前保留「上傳值 vs. 設備回讀值」的比對記錄 |

#### 通訊時序圖

```
Host                                Equipment
  │                                     │
  │─── S7F23R ─────────────────────────▶│  上傳格式化配方
  │    L:4 {                            │
  │      A "ETCH_5NM"                   │
  │      A "EQ-MODEL-1"                 │
  │      A "1.0.0"                      │
  │      L:2 {                          │
  │        L:2 { A "TEMP" L:1{F4 350.0}}│
  │        L:2 { A "TIME" L:1{U4 60}  } │
  │      }                              │
  │    }                                │
  │                                     │
  │◀─── S7F24 ──────────────────────────│  ACKC7=0（接收成功）
  │    B 0x00                           │
  │                                     │
  │  [步驟一] 先用 S7F19R 確認 PPID 已存在  │
  │                                     │
  │─── S7F19R (header only) ───────────▶│
  │                                     │
  │◀─── S7F20 ──────────────────────────│
  │    L:3 { A "ETCH_5NM" ... }         │  ← ✓ PPID 出現在清單中
  │                                     │
  │  [步驟二] S7F25R 讀回完整配方內容    │
  │                                     │
  │─── S7F25R ─────────────────────────▶│  請求讀回 ETCH_5NM
  │    A "ETCH_5NM"                     │
  │                                     │
  │◀─── S7F26 ──────────────────────────│  設備回傳完整配方
  │    L:4 {                            │
  │      A "ETCH_5NM"    ← 比對 PPID    │
  │      A "EQ-MODEL-1"  ← 比對 MDLN    │
  │      A "1.0.0"       ← 比對 SOFTREV │
  │      L:2 {           ← 逐命令比對    │
  │        L:2 { A "TEMP" L:1{F4 350.0}}│  ← ✓ 與上傳值相符
  │        L:2 { A "TIME" L:1{U4 60}  } │  ← ✓ 與上傳值相符
  │      }                              │
  │    }                                │
  │                                     │
  │  比對通過 → 配方驗證完成，可繼續換線  │
```

#### C# secs4net 實作：S7F23R 上傳 + S7F19R 存在確認 + S7F25R Round-trip 驗證

```csharp
// OutboundCommands.cs（Host 端）
public static async Task<bool> UploadAndVerifyRecipeAsync(
    ISecsGem secsGem,
    string ppid,
    string mdln,
    string softrev,
    IReadOnlyList<(string CCode, IReadOnlyList<string> Params)> commands,
    CancellationToken ct = default)
{
    // Step 1: S7F23R — 上傳格式化配方
    await SendS7F23AndWaitS7F24Async(secsGem, ppid, mdln, softrev, commands, ct);

    // Step 2: S7F19R — 確認 PPID 已出現在設備配方目錄中
    var ppids = await SendS7F19AndWaitS7F20Async(secsGem, ct);
    if (!ppids.Contains(ppid, StringComparer.Ordinal))
    {
        Console.WriteLine($"[RoundTrip] FAIL: PPID \"{ppid}\" not found in directory after upload.");
        return false;
    }
    Console.WriteLine($"[RoundTrip] Step1 OK: PPID \"{ppid}\" confirmed in S7F20 directory.");

    // Step 3: S7F25R — 讀回完整配方，逐欄位比對
    var request = new SecsMessage(7, 25, replyExpected: true, Item.A(ppid));
    SecsMessage reply = await secsGem.SendAsync(request, ct);

    var body = reply.SecsItem;
    if (body is null || body.Count < 4)
    {
        Console.WriteLine($"[RoundTrip] FAIL: S7F26 body is empty or malformed.");
        return false;
    }

    string replyPpid   = body[0].GetValue<string>();
    string replyMdln   = body[1].GetValue<string>();
    string replySoftrev = body[2].GetValue<string>();

    // 比對 PPID（空字串 = 設備回報找不到此配方）
    if (!string.Equals(replyPpid, ppid, StringComparison.Ordinal))
    {
        Console.WriteLine($"[RoundTrip] FAIL: PPID mismatch. Expected=\"{ppid}\" Got=\"{replyPpid}\"");
        return false;
    }

    // 比對 MDLN / SOFTREV（確認設備型號相容）
    if (!string.Equals(replyMdln, mdln, StringComparison.Ordinal) ||
        !string.Equals(replySoftrev, softrev, StringComparison.Ordinal))
    {
        Console.WriteLine(
            $"[RoundTrip] WARN: MDLN/SOFTREV mismatch. " +
            $"Expected=({mdln},{softrev}) Got=({replyMdln},{replySoftrev})");
        // WARN 而非 FAIL：設備可能以自己的版本號正規化
    }

    // 比對命令清單（CCODE + PPARM 逐一比對）
    var cmdList = body[3];
    if (cmdList.Count != commands.Count)
    {
        Console.WriteLine(
            $"[RoundTrip] FAIL: command count mismatch. Expected={commands.Count} Got={cmdList.Count}");
        return false;
    }

    for (int i = 0; i < cmdList.Count; i++)
    {
        string gotCCode = cmdList[i][0].GetValue<string>();
        if (!string.Equals(gotCCode, commands[i].CCode, StringComparison.Ordinal))
        {
            Console.WriteLine($"[RoundTrip] FAIL: CCODE[{i}] mismatch. Expected=\"{commands[i].CCode}\" Got=\"{gotCCode}\"");
            return false;
        }
        var gotParams = cmdList[i][1];
        if (gotParams.Count != commands[i].Params.Count)
        {
            Console.WriteLine($"[RoundTrip] FAIL: CCODE[{i}] param count mismatch.");
            return false;
        }
        for (int j = 0; j < gotParams.Count; j++)
        {
            string gotPparm = gotParams[j].GetValue<string>();
            if (!string.Equals(gotPparm, commands[i].Params[j], StringComparison.Ordinal))
            {
                Console.WriteLine(
                    $"[RoundTrip] FAIL: CCODE[{i}] PPARM[{j}] mismatch. " +
                    $"Expected=\"{commands[i].Params[j]}\" Got=\"{gotPparm}\"");
                return false;
            }
        }
    }

    Console.WriteLine($"[RoundTrip] PASS: PPID \"{ppid}\" round-trip verified ({commands.Count} commands).");
    return true;
}
```

#### MDLN / SOFTREV 不符時的處理策略

| 情況 | 建議處理 |
|------|----------|
| MDLN 不符 | **FAIL** — 配方可能為其他機型所設計，繼續使用有安全風險 |
| SOFTREV 次版本不同（如 `1.0.0` vs `1.0.1`） | **WARN** — 通常設備韌體升級後的正規化，記錄後可繼續 |
| CCODE 數量或名稱不符 | **FAIL** — 配方在設備端被修改或截斷，必須重新上傳 |
| PPARM 數值輕微差異（浮點正規化） | **WARN** — 考慮使用容差比對（`Math.Abs(a - b) < epsilon`）而非字串比對 |

> **提示：** S7F26 中的 `PPARM` 型別在 secs4net 中可能以不同 `SecsFormat` 儲存（`F4`、`U4`、`A` 等）。若上傳時使用 `Item.F4(350.0)` 但讀回為 `A "350.0"`，代表設備做了型別轉換，此時應改用 `GetValue<float>()` 後再做數值比對，而非直接字串比較。

### S7F19R 在 GEM Process Program Management 流程中的位置

```
完整 GEM 配方管理工作流程：

  1. [Host] S7F19R ──────▶ [Equipment]    ← 查詢現有配方清單（本文件）
            S7F20 ◀──────                 ← 回傳 L:n { PPID }

  2. [Host] S7F23R ─────▶ [Equipment]    ← 上傳新格式化配方
            S7F24  ◀─────                ← ACKC7（0=OK, 1~6=錯誤）

  3. [Host] S7F25R ─────▶ [Equipment]    ← 讀回配方做 round-trip 驗證
            S7F26  ◀─────                ← L:4 { PPID, MDLN, SOFTREV, L:c }

  4. [Host] S7F17R ─────▶ [Equipment]    ← 刪除指定配方
            S7F18  ◀─────                ← ACKC7

  5. [Equip] S7F7R ──────▶ [Host]        ← 物料進站，查詢 MID 對應配方
             S7F8  ◀──────               ← L:2 { PPID, MID }
```

S7F19R 是整個流程的**入口**：Host 先查清單，再決定後續的上傳、讀回、刪除動作。
