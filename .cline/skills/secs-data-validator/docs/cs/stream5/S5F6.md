# S5F6 - List Alarm Data

**Direction:** Sent by Equipment Only

**Reply to:** [S5F5R](S5F5R.md) — List Alarms Request

---

| **S5F6** | List Alarm Data | Sent by Equipment Only |

## 目的與用意

S5F6 是 SECS-II Stream 5（Exception Handling / 警報管理）中的 **查詢回覆訊息**。當 Host 想瞭解設備上某些（或全部）警報的定義與狀態時，會先發送 [S5F5R](S5F5R.md)（List Alarms Request），指定要查詢的 Alarm ID 清單；Equipment 收到後以此 S5F6 回覆對應的 alarm 資料。

### 使用情境

| 情境 | 說明 |
|------|------|
| **設備連線初始化** | Host 連線後，用 S5F5R 查詢全部 alarm（`U4:0`），再用 S5F6 取得完整 alarm 清單，建立本地 alarm 資料庫。 |
| **特定 alarm 查詢** | Host 收到 S5F1R（Alarm Report Send）後，想瞭解該 alarm 的完整定義，用 S5F5R 指定 ALID 查詢，以 S5F6 取得結果。 |
| **UI 畫面刷新** | 操作員在 Host 端警報管理畫面選擇「重新整理」，Host 發 S5F5R 取得最新 alarm 清單（S5F6）。 |

### 與其他 Stream 5 訊息的關係

```
Host                                        Equipment
 │                                             │
 │──── S5F5R (List Alarms Request) ──────────→│  Host 指定 ALID 或查全部
 │                                             │
 │←─── S5F6  (List Alarm Data) ──────────────│  Equipment 回覆 alarm 資料
 │                                             │
 │──── S5F7R (List Enabled Alarm Request) ───→│  Host 查詢「已啟用」的 alarm
 │←─── S5F8  (List Enabled Alarm Data) ──────│  只回傳 enabled 的 alarm
 │                                             │
 │←─── S5F1R (Alarm Report Send) ────────────│  Equipment 主動上報 alarm 事件
 │──── S5F2  (Alarm Report Ack) ─────────────→│  Host 確認收到
```

> **關鍵差異：** S5F6 回傳的是 **符合 S5F5R 查詢條件的所有 alarm**（含已啟用和已停用）；S5F8 只回傳 **目前已啟用（enabled）** 的 alarm。兩者資料格式完全相同（都是 `L:n {L:3 ALCD ALID ALTX}`），但語意不同。S5F1R 則是設備主動推送的即時警報通知，結構為單筆 `L:3`（無外層 list）。

## Format (TSN)

```
{L:n
{L:3
[ALCD](http://hume.com/secs/items.html#ALCD)
[ALID](http://hume.com/secs/items.html#ALID)
[ALTX](http://hume.com/secs/items.html#ALTX)
}}
```

### 資料格式詳解

#### 外層 `L:n` — Alarm 資料清單

- **型態：** `L:n`（SECS-II List，長度 `n` 可變）
- **`n` 的意義：** 符合 S5F5R 查詢條件的 alarm 筆數。
  - 若 S5F5R 傳送 `U4:0`（查全部），`n` = 設備定義的全部 alarm 數量。
  - 若 S5F5R 指定 3 個 ALID，`n` 最大為 3（若某 ALID 不存在則不含在回覆中）。
  - 若沒有符合條件的 alarm，`n = 0`（回覆 `L:0`，空 list）。

#### 內層 `L:3` — 單筆 Alarm 資料

每筆 alarm 固定為 3 個元素的 list：

| 順序 | 名稱 | SECS-II 型態 | TSN 表示 | 說明 |
|------|------|-------------|---------|------|
| 1 | ALCD | Binary (1 byte) | `B:1` | Alarm Code — 警報狀態碼 |
| 2 | ALID | Unsigned Int 4-byte | `U4:1` | Alarm ID — 警報識別碼 |
| 3 | ALTX | ASCII string | `A:n` (n ≤ 120) | Alarm Text — 警報描述文字 |

#### ALCD — Alarm Code（`B:1`）

`B:1` 是 SECS-II 的 Binary 格式，佔 1 byte（8 bits），以無號整數解讀：

| ALCD 值 | 二進位 | 意義 | 說明 |
|---------|--------|------|------|
| `0` | `0000 0000` | Cleared, Category 0 | 警報已清除，無分類 |
| `1` | `0000 0001` | Cleared, Category 1 | 警報已清除，第 1 類 |
| `2` | `0000 0010` | Cleared, Category 2 | 警報已清除，第 2 類 |
| `…` | `…` | Cleared, Category N | 低 7 bits 為分類代碼 |
| `128` | `1000 0000` | **Set**, Category 0 | **警報觸發中**，無分類 |
| `129` | `1000 0001` | **Set**, Category 1 | **警報觸發中**，第 1 類 |
| `130` | `1000 0010` | **Set**, Category 2 | **警報觸發中**，第 2 類 |

> **判斷規則：** bit 7（最高位）= 1 → alarm is **SET**（觸發中）；bit 7 = 0 → alarm is **CLEAR**（已清除）。低 7 bits 的分類用途已被 SEMI 棄用（deprecated），實務上多數設備只使用 `0`（clear）和 `128`（set），但 Schmoll S80 等設備會使用 `2` / `130` 等帶分類值。

#### ALID — Alarm ID（`U4:1`）

`U4:1` 是 SECS-II 的 Unsigned 4-byte Integer 格式，佔 4 bytes，範圍 0 ~ 4,294,967,295：

- 每個 ALID 對應設備上一個唯一定義的警報種類（例如「溫度過高」、「壓力異常」、「緊急停止」等）。
- ALID 由設備廠商定義，Host 端通常會建立一個 ALID → 描述的對照表。
- 在 S5F6 中，ALID 值**不會重複**（每筆 alarm 對應一個唯一的 ALID）。

> **Schmoll S80 的 ALID 慣例：** ALID 末 3 碼會出現在 ALTX 的括號前綴中。例如 ALID=`1000004` 對應 ALTX=`(004) Loader: Emergency stop right active`，其中 `(004)` 就是 ALID 的末 3 碼。

#### ALTX — Alarm Text（`A:n`）

`A:n` 是 SECS-II 的 ASCII 字串格式，`n` 為字串長度（最大 120 字元）：

- 描述警報的人類可讀文字，供操作員辨識。
- 長度上限原為 40 字元（SEMI E5 舊版），後提升至 120 字元。
- 不同設備廠商的 ALTX 格格不入：
  - **Schmoll S80：** 使用 `(NNN)` 前綴 + 空格 + 描述文字的格式（如 `(004) Loader: Emergency stop right active`）。
  - **一般設備：** 直接使用純描述文字（如 `Temperature High`）。

### 格式層級圖

```
S5F6 data
 └── L:n                          ← 外層 list，n = alarm 筆數
      ├── L:3                     ← 第 1 筆 alarm
      │    ├── B:1  <ALCD>        ← Alarm Code (1 byte binary)
      │    ├── U4:1 <ALID>        ← Alarm ID   (4-byte unsigned int)
      │    └── A:n  <ALTX>        ← Alarm Text (ASCII, max 120 chars)
      ├── L:3                     ← 第 2 筆 alarm
      │    ├── B:1  <ALCD>
      │    ├── U4:1 <ALID>
      │    └── A:n  <ALTX>
      └── ...                     ← 更多 alarm（如有）
```

### 範例

#### 範例 1：查詢全部 alarm，回傳 2 筆

對應 S5F5R 發送 `U4:0`（查全部）：

| TSN | 意義 |
|-----|------|
| `L:2 {L:3 {B:1 128} {U4:1 1001} {A:20 Temp High}} {L:3 {B:1 0} {U4:1 1002} {A:18 Pressure OK}}` | 2 筆 alarm：ALID=1001 正在觸發（ALCD=128≥128）、ALID=1002 已清除（ALCD=0<128） |

逐欄解析：

| 筆數 | ALCD | ALCD 意義 | ALID | ALTX |
|------|------|----------|------|------|
| 第 1 筆 | `128` | Set（觸發中） | `1001` | `Temp High` |
| 第 2 筆 | `0` | Clear（已清除） | `1002` | `Pressure OK` |

#### 範例 2：查詢特定 ALID，回傳 1 筆

對應 S5F5R 發送 `U4:1 1000004`（只查 ALID=1000004）：

| TSN | 意義 |
|-----|------|
| `L:1 {L:3 {B:1 2} {U4:1 1000004} {A:39 (004) Loader: Emergency stop right active}}` | 1 筆 alarm：ALID=1000004 已清除（ALCD=2，分類 2） |

#### 範例 3：查詢不存在的 ALID，回傳空清單

對應 S5F5R 發送 `U4:1 9999999`（查一個不存在的 ALID）：

| TSN | 意義 |
|-----|------|
| `L:0` | 沒有符合條件的 alarm（空 list） |

#### 範例 4：多筆 alarm，混合 Set/Clear 狀態

| TSN | 意義 |
|-----|------|
| `L:3 {L:3 {B:1 130} {U4:1 1000004} {A:39 (004) Emergency stop right active}} {L:3 {B:1 128} {U4:1 1000199} {A:45 (199) External feed hold active}} {L:3 {B:1 2} {U4:1 3000227} {A:40 (C227) Not enough tools}}` | 3 筆：前 2 筆觸發中，第 3 筆已清除 |

### 實機案例（Schmoll KA01 S80 鑽孔機）

S5F5 查詢 ALID=`1000004` 後 Equipment 回覆：

```xml
<SECSMessage s="5" f="6" direction="E to H" replyBit="false" txid="5001001" time="2026-03-19T10:00:00.020Z" deviceID="0">
    <SECSData>
        <LST><LST><BIN>2</BIN><UI4>1000004</UI4><ASC>(004) Loader: Emergency stop right active</ASC></LST></LST>
    </SECSData>
</SECSMessage>
```

**XML ↔ TSN 對照：**

| XML 元素 | SECS-II 型態 | TSN 表示 | 值 |
|----------|-------------|---------|-----|
| `<LST>...</LST>`（外層） | List | `L:1` | 包含 1 筆 alarm |
| `<LST>...</LST>`（內層） | List | `L:3` | 固定 3 元素 tuple |
| `<BIN>2</BIN>` | Binary 1-byte | `B:1 2` | ALCD = 2（Cleared, Cat.2） |
| `<UI4>1000004</UI4>` | Unsigned Int 4-byte | `U4:1 1000004` | ALID = 1000004 |
| `<ASC>(004) Loader: Emergency stop right active</ASC>` | ASCII | `A:39 (004) Loader: Emergency stop right active` | ALTX 描述文字 |

**重點：**
- ALCD=`2`（< 128）表示此警報處於 **cleared** 狀態。即使緊急停止的物理狀況已排除，該警報紀錄仍可被 S5F5 查詢到——這是「警報定義」與「警報狀態」的分離：S5F6 回傳的是 alarm 的**定義資料**（含目前狀態），不只是正在觸發的 alarm。
- `replyBit="false"` 表示此為 reply 訊息（不需要再回覆）。
- ALTX 的 `(004)` 前綴對應 ALID 末 3 碼 `1000`**`004`**，是 Schmoll S80 的 ALTX 命名慣例。

## C# 接收處理程式

以下 C# 程式碼是 **Host 端** 發送 S5F5 後，解析 Equipment 回覆 S5F6 的邏輯，使用 `secs4net` 函式庫：

### 處理流程

1. **發送 S5F5 並等待回覆** — 呼叫 `sp.SendSecsMsg(5, 5, true, sendData, true)` 發送查詢請求。
2. **檢查回傳結果** — `rcr.rc != 0` 表示通訊失敗，直接跳出。
3. **解析外層 list** — 用 `SecsPort.ListSplit()` 拆解 `L:n`，`list0[0]` 為 `"L:n"`，`list0[1..n]` 為各組 alarm 資料。
4. **迴圈解析每組 alarm** — 對每個 `list0[n]` 再次呼叫 `ListSplit()` 得到 `list1`，驗證 `list1[0] == "L:3"`，然後依序解析 `ALCD`（`sp.BinToInt()`）、`ALID`（U4 字串）、`ALTX`（ASCII 字串）。
5. **錯誤檢查** — 任何一組解析失敗會設 `ok = false` 並 break 內層迴圈，外層再 break。

### 資料型態對應（SECS-II ↔ C#）

| SECS-II 型態 | TSN 標記 | C# 變數型態 | 解析方式 |
|-------------|---------|------------|---------|
| Binary (1 byte) | `B:1` | `int` | `sp.BinToInt(list2[1])` |
| Unsigned Int 4-byte | `U4:1` | `string` | 直接取 `list2[1]`（字串形式的數字） |
| ASCII string | `A:n` | `string` | 直接取 `list2[1]`（純文字） |
| List | `L:n` | `string[]` | `SecsPort.ListSplit()` 拆解為陣列 |

```csharp
    // S5F6 C# Parse reply
    SecsHost sp=(SecsHost)your_SecsHost;
    RcResult rcr = sp.SendSecsMsg(5, 5, true, sendData, true);
    string TSN_data;
    bool ok=true;
    while (ok) {   // break out of loop on error
        if (rcr.rc != 0) {ok=false; break;}
        TSN_data = rcr.result;
        // expect L:n {L:3 ALCD ALID ALTX}
        // variables for data items and parsing
        int ALCD;  	// B:1 (always)  alarm code byte, >= 128 alarm is set, bit field use is deprecated
        string ALID;  	// U4:1 (varies)  Alarm type ID
        string ALTX;  	// A:120 (always)  alarm text, the length limit was recently raised from 40
        string [] list0;
        list0 = SecsPort.ListSplit(TSN_data);
        if (list0.Length < 1) { ok=false; break; }
        int n;
        for(n=1; n < list0.Length; n++) {
            string [] list1;
            list1 = SecsPort.ListSplit(list0[n]);
            if (list1.Length != 4 || list1[0] != "L:3") { ok=false; break; }
            string [] list2;
            list2 = SecsPort.ListSplit(list1[1]);
            if (list2.Length < 1) { ok = false; break; }
            if (list2[0] != "B:1") { ok = false; break; }
            ALCD = sp.BinToInt(list2[1]);
            list2 = SecsPort.ListSplit(list1[2]);
            if (list2.Length < 1) { ok = false; break; }
            if (list2.Length == 2) { ALID = list2[1];} else { ALID = ""; }
            list2 = SecsPort.ListSplit(list1[3]);
            if (list2.Length < 1) { ok = false; break; }
            if (!list2[0].StartsWith("A:")) { ok = false; break; }
            if (list2.Length == 2) { ALTX = list2[1];} else { ALTX = ""; }
            }
        if (!ok) break;
        return;  // parsed ok
        } // end while(ok)
```

### 解析邏輯逐步說明

```
Step 1: SecsPort.ListSplit(TSN_data)  →  list0 = ["L:n", "inner1", "inner2", ...]
                                                          ↑ index 從 1 開始（跳過 "L:n" 本身）

Step 2: 對每個 list0[n] 呼叫 ListSplit   →  list1 = ["L:3", "ALCD_raw", "ALID_raw", "ALTX_raw"]
                                                          ↑ 驗證長度 == 4 且 tag == "L:3"

Step 3: 解析各欄位
  ALCD: ListSplit(list1[1]) → ["B:1", "128"]  → sp.BinToInt("128") = 128
  ALID: ListSplit(list1[2]) → ["U4:1", "1001"] → "1001"（字串）
  ALTX: ListSplit(list1[3]) → ["A:20", "Temp High"] → "Temp High"（字串）
```

> **注意：** `ListSplit()` 回傳的陣列中，`[0]` 是型態標記（如 `"B:1"`, `"U4:1"`, `"A:20"`），`[1]` 才是實際值。若型態標記存在但值為空（如 `<UI4 />`），則陣列長度為 1，程式碼以 `list2.Length == 2` 判斷並給予預設空字串。
