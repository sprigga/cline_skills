# S5F2 - Alarm Report Ack

**Direction:** Sent by Host Only

---

| **S5F2** | Alarm Report Ack | Sent by Host Only |

## 用意與目的

S5F2 是 Host 對設備 S5F1R（Alarm Report Send）的確認回覆。每當設備主動上報警報狀態（SET 或 CLEAR），Host 必須在 T3 時間內以 S5F2 回應，告知設備是否成功接收。

### 在 GEM 標準中的角色

```
Equipment                          Host
   │                                 │
   │  S5F1R (Alarm Report Send)  ──▶ │  ← 設備主動推送警報
   │                                 │    Host 收到後必須立即回覆
   │  S5F2  (Alarm Report Ack)   ◀── │  ← Host 確認收到（ACKC5=0）
   │                                 │
```

S5F2 是整個警報通知流程的「閉環確認」。若 Host 未在 T3 內回覆，設備端視為通訊失敗，可能記錄 T3 Timeout 事件或觸發重連。

### 訊息結構特點

S5F2 是 SECS-II 中少數**不使用 List 包裝**的訊息：根 Item 直接是 `B:1`，而非 `L:1 { B:1 }`。這與大多數包含多個 data item 的訊息不同，需特別注意，不可額外加一層 List。

---

## Format (TSN)

```
[ACKC5](http://hume.com/secs/items.html#ACKC5)
```

### Format Explanation

- **`[ACKC5]`** — Acknowledge Code for Stream 5，`B:1` 格式（1 byte binary）。
  - `0` = OK（確認收到 alarm report）
  - 其他值 = 錯誤或拒絕

這是一個**簡單的單一 data item** 回覆，Host 用來回覆 Equipment 發送的 S5F1R（Alarm Report Send），告知是否成功接收。

### ACKC5 數值對照表

| 值 | 意義 |
|----|------|
| `0` (`0x00`) | OK — 成功接收 alarm report |
| 非零 | 拒絕或錯誤（GEM 標準未細分，Host 端記錄為 rejected） |

> **實務說明：** GEM 標準中，Host 幾乎總是回覆 `ACKC5=0`。若 Host 解析 S5F1 失敗，通常不回覆 S5F2（導致設備 T3 timeout），而非回覆非零值。

### 範例

| 傳送內容 | 意義 |
|---------|------|
| `B:1 0` | 確認收到 alarm report（OK） |
| `B:1 1` | 拒絕或錯誤 |

## C# 發送程式碼（secs4net v2）

S5F2 不是一個「接收」的訊息，而是 Host 在處理 S5F1R 時**主動建構並發送**的回覆。以下程式碼展示如何在 `S5F1Handler` 內部建構並發送 S5F2：

### 建構與發送流程

1. **建立 `SecsMessage(5, 2)`** — 指定 Stream=5、Function=2。
2. **設定 `SecsItem = B(0x00)`** — 根 Item 直接為 `B:1`，ACKC5=0 表示成功。
3. **呼叫 `TryReplyAsync`** — 透過 `PrimaryMessageWrapper` 回覆，框架自動關聯對應的 Transaction ID。

```csharp
// 在 S5F1Handler.HandleAsync 內部，回覆 S5F2
private static async Task ReplyAckAsync(PrimaryMessageWrapper wrapper, CancellationToken ct)
{
    using var s5f2 = new SecsMessage(5, 2)
    {
        Name     = "AlarmReportAck",
        SecsItem = B(0x00), // ACKC5 = 0 (accepted)
                            // 注意：直接是 B:1，不是 L:1 { B:1 }
    };
    bool sent = await wrapper.TryReplyAsync(s5f2, ct);
    Console.WriteLine($"[TX] S5F2 ACK → {(sent ? "OK" : "Already replied")}");
}
```

> **Item 工廠方法：** `B(0x00)` 來自 `using static Secs4Net.Item;`，等同於 `Item.B((byte)0x00)`，建立一個 `B:1` 格式的單 byte binary Item。

### 完整的 S5F1Handler 上下文

S5F2 由 `S5F1Handler` 在接收 S5F1R 後觸發，請參閱 [S5F1R.md](S5F1R.md) 的「C# 接收處理程式（secs4net v2）」章節取得完整實作。

## 舊版 API 程式碼（secs4net v1 / SecsPort）

以下 C# 程式碼使用舊版 `secs4net`（`SecsPort` 時代）API，展示 **Equipment 端** 發送 S5F1 後解析 S5F2 回覆的邏輯。**不適用於本專案**，僅供對照參考。

### 處理流程

1. **發送 S5F1 並等待回覆** — 呼叫 `sp.SendSecsMsg(5, 1, true, sendData, true)` 發送 alarm report，其中 `true` 表示需要等待回覆。
2. **檢查回傳結果** — `rcr.rc != 0` 表示通訊失敗（例如 timeout），直接跳出。
3. **解析 ACKC5** — 用 `SecsPort.ListSplit()` 拆解回覆資料，驗證 type tag 為 `B:1`，再用 `sp.BinToInt()` 將 binary 轉為整數。
4. **判斷結果** — `ACKC5 == 0` 表示成功。

```csharp
    // S5F2 C# Parse reply
    SecsPort sp=(SecsPort)your_SecsPort
    RcResult rcr = sp.SendSecsMsg(5, 1, true, sendData, true);
    string TSN_data;
    bool ok=true;
    while (ok) {   // break out of loop on error
        if (rcr.rc != 0) {ok=false; break;}
        TSN_data = rcr.result;
        // expect ACKC5
        // variables for data items and parsing
        int ACKC5;  	// B:1 (always)  acknowledge code, 0 ok
        string [] list0;
        list0 = SecsPort.ListSplit(TSN_data);
        if (list0.Length < 1) { ok = false; break; }
        if (list0[0] != "B:1") { ok = false; break; }
        ACKC5 = sp.BinToInt(list0[1]);
        return;  // parsed ok
        } // end while(ok)
```

## 實測 Log 案例（KA01 Schmoll 鑽孔機，2026-04-14）

以下為 S5F1R / S5F2 的實際 XML log，可對照確認 Host 回覆結構：

```xml
<!-- 設備發送 S5F1R：ALCD=0x02，ALID=5003，ALTX="2" -->
<SECSMessage s="5" f="1" direction="E to H" replyBit="true"
             txid="186" time="2026-04-14T13:01:10.325Z" deviceID="0">
    <SECSData>
        <LST>
            <BIN>2</BIN>
            <UI4>5003</UI4>
            <ASC>2</ASC>
        </LST>
    </SECSData>
</SECSMessage>

<!-- Host 回覆 S5F2：ACKC5=0（OK），注意根節點直接是 BIN，無 LST 包裝 -->
<SECSMessage s="5" f="2" direction="H to E" replyBit="false"
             txid="186" time="2026-04-14T13:01:10.341Z" deviceID="0">
    <SECSData><BIN>0</BIN></SECSData>
</SECSMessage>
```

**觀察重點：**

- `replyBit="true"` 在 S5F1R 上 — 設備要求 Host 回覆
- `replyBit="false"` 在 S5F2 上 — 回覆訊息不需再回覆
- S5F2 的 `<SECSData>` 內直接是 `<BIN>0</BIN>`，沒有 `<LST>` 包裝，對應格式規範的單一 `B:1` Item
- 回覆延遲：`13:01:10.341Z - 13:01:10.325Z = 16ms`，遠低於 T3=45s 限制
